# @idefy/core — Renderers

Подсистема рендеринга — это **pluggable архитектура**, отделённая от форматтера. Контракт спроектирован так, чтобы реальный ASCII-рендерер (или будущие SVG/Mermaid рендереры) встал на место заглушки без изменений по контуру.

## Контракт рендерера

```
Renderer {
    id: RendererId          // 'ascii' для текущей итерации; будут 'svg', 'mermaid' и др.
    render(project: IdefProject): RenderResult
}

RenderResult {
    sidecars: ReadonlyMap<string, string>   // ключ — путь файла, к которому привязан sidecar
                                            // (т.е. путь .idef0 файла, не .idef0.ascii)
                                            // значение — контент sidecar'а как строка
    diagnostics: Diagnostic[]               // ошибки/предупреждения рендеринга
}
```

`render` принимает целый `IdefProject` (не один файл), потому что в общем случае рендеринг листа может требовать контекста: глобальные туннели из `A-0`, родительский интерфейс и т.д. Заглушка эти зависимости не использует, реальные рендереры — будут.

**Расширение sidecar-файла** — на стороне caller'а, не рендерера. Renderer не знает, что результат для `A1.idef0` будет записан в `A1.idef0.ascii`. Маппинг ключа `RenderResult.sidecars` (`path/to/A1.idef0`) на путь sidecar'а (`path/to/A1.idef0.ascii`) делает [`@idefy/loader`](../../loader/spec/COMPONENT.md) или [`@idefy/vscode`](../../vscode/spec/COMPONENT.md). Расширение `.ascii` зашито в `@idefy/loader`.

## Renderer registry

```
RendererRegistry {
    register(renderer: Renderer): void
    get(id: RendererId): Renderer | null
    listAvailable(): RendererId[]
}
```

`@idefy/core` экспортирует фабрику `createRendererRegistry()`, возвращающую готовый registry с зарегистрированным `ascii` рендерером.

В текущей итерации единственный потребитель registry — пайплайн форматтера, который после форматирования DSL вызывает `registry.get('ascii').render(project)` и пишет полученные sidecar'ы. Это решение caller'а (`@idefy/vscode` оркеструет последовательность), не самого рендерера.

## ASCII-рендерер — текущая итерация (заглушка)

**Поведение:**

- Для каждого `.idef0` файла в `IdefProject.files`, кроме `A-0.*.idef0` (контекст не имеет визуального листа), возвращает текстовую **сводку идентификаторов** активити: заголовок с ID и именем активити, затем по строке на роль (I / C / M / O / X / T) с перечислением встретившихся ID в той роли, и в конце — список дочерних активити. Пример:

  ```
  # IDEFy ASCII placeholder for A1 "Помол зерна"
    I: I1
    C: C1
    M: M1 M2
    O: O1
    X: X11
    children: A11 A12
  ```

- `RenderResult.diagnostics` пустой.

**Это намеренная заглушка**, оправдывающая интерфейс рендерера и форматтерный пайплайн целиком, без необходимости проектировать алгоритм раскладки в ASCII в этой итерации. Сводка содержит **реальные DSL-идентификаторы** из проекта — это обеспечивает наглядную проверку UI-пайплайна `@idefy/vscode` (ролевая раскраска, PNG-экспорт, FS lifecycle): WebView сканирует контент sidecar'а тем же лексическим скенером, что и DSL, и красит каждый ID по ролевой палитре.

**Когда `[TODO]` снимется:** заглушка из этого файла удаляется, поведение заменяется на реальный рендер. Интерфейс `Renderer` и `RendererRegistry` не меняется.

## Будущие рендереры (forward direction)

- `svg` — SVG-рендер диаграммы. Тот же интерфейс, другой формат вывода. Sidecar `<name>.idef0.svg`.
- `mermaid` — Mermaid-разметка. Sidecar `<name>.idef0.mmd` или вставка в Markdown.

Эти рендереры **не обсуждаются** в текущей итерации; перечислены здесь только как обоснование того, почему интерфейс — pluggable.

## Invariants

- Рендерер не выбрасывает исключений на любом `IdefProject` (включая частично собранные). Все ошибки — через `diagnostics`.
- Рендерер не зависит от файловой системы и не делает I/O. Возвращает строки; запись на диск — задача caller'а.
- Рендерер read-only относительно `IdefProject` — не мутирует входные данные.
