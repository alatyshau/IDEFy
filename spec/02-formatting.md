# Форматирование и стиль IDEF0 DSL

Этот документ описывает стиль написания и форматирования исходного кода `.idef0`, который не является строгим для валидатора, но используется для поддержания читаемости и применяется плагином-форматтером (например, в VS Code) как строгое ТЗ.

## Граничные стрелки

1. Объявления граничных стрелок (интерфейс активности) всегда размещаются строго по одной стрелке на каждую строку.
2. Между блоком граничных стрелок и списком функциональных блоков декомпозиции должна быть ровно **одна пустая строка**.
3. Внутри секции стрелки сортируются по группам в порядке `I → C → M → O → T`. Внутри каждой группы стрелки сортируются по тому же правилу, что и в списках consumed (см. «Сортировка стрелок внутри групп» ниже): сначала плоские ID, затем bracket-формы. Туннели идут в конце отдельной плоской группой `T*` без bracket-форм (роль туннеля определяется на месте использования внутри декомпозиции, не в boundary — см. `01-dsl.md`, «Правила для туннелей»). Это применяется и к корневому `A0`, и к декомпозированным потомкам.

Корневая активность (плоские формы; туннель в конце boundary только если внутри декомпозиции `A0` его кто-то использует):

```idef0
activity A0 "Работа с Креатором" {
    I1 "set process project"
    C1 "PSL"
    O1 "show named process project"
    T1 "Шум в офисе"

    A1 "Проектирование САПР" ...
```

Декомпозированный потомок (с bracket-формами для sibling-X и own-described X parent'а; туннели остаются плоскими; ID/описания эхо-наследуются от owner-уровня — см. `01-dsl.md`, «Согласованность интерфейса между уровнями»):

```idef0
activity A1 "Помол зерна" {
    I1 "Свежеобжаренное зерно"
    C1 "Рецепт"
    M1 "Бариста"
    M2 "Кофемолка"
    O[X11] "Смолотый кофе"
    T2 "Рыночные условия"

    A11 "Разрушение оболочки" ...
```

## Функциональные блоки

Блоки внутри декомпозиции всегда **сортируются по их ID** (от `A1` до `A9`, `Aa`..`Az`, затем внуки по той же логике). Пользователь управляет порядком блоков, задавая им нумерацию. Если пользователь изменит номер (например, `A2` на `A5`), авто-форматтер обязан переместить строку в соответствии с сортировкой.

Для форматирования функциональных блоков предусмотрено **три режима канонического расположения**:

### Режим 1: В одну строку

Используется для коротких объявлений. Примыкающие друг к другу блоки в Режиме 1 группируются без пустых строк.

**Правила выравнивания и группировки (Табличный вид):**
1. Двоеточия `:` выравниваются в единую вертикальную колонку с помощью пробелов/табуляции для всех блоков в Режиме 1 в файле.
2. Потребляемые стрелки слева от `->` группируются строго по ролям: `I`, затем `C`, затем `M`. Форматтер создает визуальные колонки под эти группы. Если группа (например, `C`) отсутствует в данном блоке, её место остаётся пустым, чтобы выровнять соседние группы.
3. Стрелки перехода `->` выравниваются в единую вертикальную колонку по самой широкой комбинации входов в файле.

```idef0
    A1 "Поиск"         : I1,     C2, C3, C[T2], M1, M3 -> X11 "Кандидаты"
    A2 "Интервью"      : I[X11], C3,            M2, M4 -> X21 "Оценки"
    A3 "Оффер"         : I[X21], C4, C[T1],     M1, M2 -> X31 "Оффер", X32[O2]
    A4 "Оформление"    : I[X31], C1, C2,        M1, M2 -> X41[O1]
```

### Режим 2: На три строки

Используется, когда объявление не помещается на экран, но списки стрелок до и после `->` достаточно короткие. Между блоком в Режиме 2 (или 3) и соседними блоками ставится **одна пустая строка**.
- На первой строке остаётся только ID и имя блока.
- На вторую строку переносится двоеточие `:` и все потребляемые стрелки. Добавляется один уровень отступа (табуляция) спереди. Стрелки разбиваются на группы `I`, `C`, `M`, но без жестких табличных колонок.
- На третью строку переносится `->` и все производимые стрелки. Используется тот же один уровень отступа.

```idef0
    A13 "Задание входов и выходов процесса"
        : I[X112], I[X113], I[X114], C1
        -> X131 "Спецификация входов и выходов процесса"
```

### Режим 3: Многострочный с переносом стрелок

Аналогично Режиму 2, но всё, что идёт после `->`, дополнительно переносится на последующие строки (по одной производимой стрелке на строку) с еще одним дополнительным уровнем табуляции.

```idef0
    A11 "Разбор проекта процесса"
        : I1, C1
        ->
            X111 "set <process name>",
            X112 "set <input>",
            X113 "set <output>",
            X114 "kind"
```

## Produced-формы с label

При join (см. `01-dsl.md` раздел «Plug vs Socket» и `04-validator.md` правило 21) bracket-формы в produced list могут нести текстовый label:

- `X11[O1] "audio"` — plug X11 в сокет O1 c label.
- `X11[T1] "alpha"` — plug X11 в туннель T1 c label.
- `X11[X22] "left"` — plug X11 в parent's own-described X22 c label.

Форматтер сохраняет label буквально (с обычным quoting через `\"` и `\\`). При сортировке produced-list внутри блока label не участвует в ключе — сортировка идёт по plug-id (`X11`, `X12`, …). Между bracket-формой и label — один пробел; внутри `[...]` — без пробелов.

При классификации блоков на режимы 1/2/3 plug label учитывается как «значимая собственная подпись» для триггера режима 3: если в produced более одной формы с собственным описанием (включая own-described `X* "..."` и любые plug labels), включается режим 3.

## Сортировка стрелок внутри групп

Внутри каждой роли (группы `I`, `C`, `O`, `M`) стрелки **обязательно сортируются**.
Правило сортировки: цифры и буквы идут перед символом `[`. 
Это гарантирует, что собственные (короткие) стрелки уровня `I1`, `I2`, `C4` выводятся первыми, а затем идут стрелки соседей `I[X...]` и туннели `I[T...]`.

Пример правильной сортировки:
`I1, I2, I[T1], I[X11], I[X22]`


## Правила выбора режимов (Триггеры авто-форматтера)

По умолчанию для всех функциональных блоков используется **Режим 1**. Переход к более сложным режимам происходит по следующим правилам:

1. **Триггер для Режима 2:**
   Если вычисленная длина строки блока в Режиме 1 (включая все выравнивающие табуляции/пробелы) превышает **120 символов**, блок переключается в **Режим 2**.
   *Примечание: Исключение этого блока из Режима 1 пересчитывает ширину колонок для оставшихся блоков в Режиме 1 (что делает их компактнее).*

2. **Триггер для Режима 3:**
   Режим 3 включается **только** когда среди производимых стрелок (после `->`) есть **более одной стрелки с собственным описанием** (формата `X.. "Описание"`). Один own-described produced + длинная строка → достаточно Режима 2 (пример: `A13 ... -> X131 "..."` ниже). Несколько стрелок-наследников вида `X[O..]`/`X[T..]` без описаний — Режим 3 не нужен. Режим 3 используется именно когда требуется развернуть produced-список по одной стрелке на строку из-за множественных собственных описаний.


3. **Подрежим Б (Длинный список потребляемых стрелок):**
   Если блок находится в Режиме 2 или 3, и длина строки с потребляемыми стрелками (которая начинается с `: `) превышает **120 символов**, включается дополнительный перенос для левой части.
   В этом случае:
   - Двоеточие `:` остаётся на второй строке (с одним табом).
   - Каждая группа потребляемых стрелок (`I`, `C`, `M`) переносится на свою отдельную строку с дополнительным уровнем табуляции (два таба).
   - Если группа (строка) всё равно превышает 120 символов, дальнейший перенос внутри неё **не делается** (игнорируем, YAGNI).

   Пример:
   ```idef0
    A15 "Очень сложная активность с кучей входов"
        :
            I1, I2, I[X111], I[X112], I[X113], I[X114],
            C1, C2, C[X121], C[X131],
            M1, M2, M3
        -> X151[O1]
   ```



**Важно:** Перенос внутри ID стрелки, внутри ID активности или внутри строкового литерала запрещён в любом из режимов.


## Floating-комментарии (не прилипшие)

Комментарии, не прилипшие ни к одному функциональному блоку (в `activity`) или туннелю/`...A0`-ссылке (в `context A-0`), хранятся парсером в `floatingComments` верхнего уровня AST. Форматтер обрабатывает их так:

- **Pre-header floating** — комментарии, расположенные в исходнике **до** заголовка `activity`/`context` (file-level комментарии вроде `# src/idef0/.../A0.idef0`) — сохраняются в выходе формата на той же позиции: сверху файла, отделены пустой строкой от заголовка.
- **Body-level floating** — комментарии внутри тела (`{ ... }`), оказавшиеся вне зон прилипания (например, между секциями boundary и blocks, между группами в boundary-секции, перед закрывающей `}`) — **удаляются** при форматировании.

Это сознательное упрощение MVP: внутри `{}`-тела канонический вывод формата задаётся строго по правилам секций (boundary → blank → blocks) и режимов 1/2/3, и стабильного места для «болтающихся» комментариев в этом выводе нет. Если пользователю важно сохранить заметку — её надо прилепить к блоку/туннелю по правилу прилипания (см. ниже).

## Метаданные парсера для round-trip

Парсер сохраняет в AST поле `leadingBlankLine` на функциональных блоках и туннелях — флаг «перед этой группой в исходнике стояла пустая строка». Это **trivia для round-trip** (форматтер на следующем проходе видит ту же группировку, валидатор может ссылаться на изначальную раскладку), но **на канонический вывод формата `leadingBlankLine` не влияет** — пустые строки в выходе задаются исключительно правилами секций и режимов 1/2/3:

- Между блоком boundary и списком functional-блоков — ровно одна пустая.
- Между двумя Mode 1 блоками — без пустой.
- Между Mode 1 и Mode 2/3 блоками (в любую сторону) — одна пустая.
- Между двумя Mode 2/3 блоками — одна пустая.

То есть «ведущая пустая едет с группой при сортировке» — это иллюстрация классификации комментариев (см. пример ниже), а не canonical output формата. После пересортировки в выходе пустые строки задаются модовыми правилами заново.

## Комментарии при сортировке блоков

Сортировка блоков по ID (см. раздел «Функциональные блоки» выше) обязана сохранять связь комментариев `# ...` с блоками. Правило:

**Комментарий «прилипает» к блоку, если между комментарием и блоком нет пустой строки.** Группа [блок + прилипшие комментарии + ведущая пустая строка] переезжает вместе при сортировке.

Это значит:
- Комментарий **сверху** блока (выше блока без пустой строки между ними) прилипает к этому блоку.
- Комментарий **снизу** блока (ниже блока без пустой строки между ними) прилипает к этому блоку.
- Пустая строка между группой [блок + прилипшие комментарии] и следующей группой — это «ведущая пустая» следующей группы; она переезжает со следующей группой.
- Если между двумя комментариями есть пустая строка — они принадлежат разным группам.

**Пример. До форматирования:**

```idef0
A2 "what"
# раз

# два
# три
A1 "something"
```

**После форматирования:**

```idef0

# два
# три
A1 "something"
A2 "what"
# раз
```

Логика: `# раз` прилипает к `A2` снизу (нет пустой строки); `# два`/`# три` прилипают к `A1` сверху (нет пустой строки между ними и `A1`). Пустая строка, разделявшая две группы, едет с группой `A1` как ведущая.

(Примеры выше упрощены — без шапки `activity`, это иллюстрация именно правила обработки комментариев, не полного синтаксиса.)

## Контекстный файл `A-0`

Контекстный файл (`A-0.*.idef0`) содержит шапку `context A-0 "..."`, плоский список туннелей `T*` и опциональную ссылку `...A0`. Правила форматирования:

1. Декларации туннелей — по одной на строку (как и граничные стрелки в `activity`).
2. Туннели внутри блока **сортируются по ID** (правило сортировки то же, что для стрелок внутри ролей-групп).
3. Между блоком туннелей и опциональной ссылкой `...A0` — ровно **одна пустая строка**.
4. Если ссылки `...A0` нет — туннели идут до закрывающей `}` без trailing пустой строки.
5. Комментарии в контекстном файле подчиняются тому же правилу прилипания, что и в `activity` (см. предыдущий раздел): прилипают к ближайшему элементу без пустой строки, переезжают с элементом при сортировке.

Пример канонически отформатированного `A-0.idef0`:

```idef0
context A-0 "Контекст приготовления кофе" {
    T1 "Шум в офисе"
    T2 "Municipal Bylaw"

    ...A0
}
```

## Текст после закрывающей `}` шапки

Синтаксис DSL разрешает произвольный текст после закрывающей `}` шапки `activity` или `context` (см. `01-dsl.md`, разделы «Базовая структура файла» и «Комментарии»). Этот текст работает как неформальный подвал.

**Контракт форматтера:** при форматировании любой текст после закрывающей `}` шапки **удаляется**. Форматтер работает поверх AST, который не сохраняет произвольный raw-текст вне шапки.

Это сознательный отказ: формат вызывается по явной команде, пользователь знает, что нажимает. Если нужны устойчивые заметки про активность — используйте `#`-комментарии **внутри** шапки (они прилипают к блокам и переживают форматирование) либо отдельные файлы рядом.
