# Компилятор — спецификация поведения

## Назначение

Компилятор читает `.idef0` файлы проекта, собирает из них единую модель и проверяет её на соответствие правилам нотации. Выдаёт ошибки и предупреждения.

В первой версии **не генерирует визуальные представления** (SVG, DOT и т.п.) — это out of scope.

## Входы

Компилятор сканирует исключительно каталог `src/idef0/` рекурсивно и собирает все файлы `*.idef0` независимо от того, в какой подпапке они лежат. Файлы вне `src/idef0/` компилятору недоступны и в модель не попадают.

## Парсинг

Для каждого файла:

1. Извлекает ID процесса из имени файла (часть до первой точки).
2. Разбирает содержимое с учётом логических объявлений: объявления стрелок и шапки `process`/`context` остаются однострочными, а объявления функциональных блоков процесса могут занимать несколько физических строк согласно правилам DSL.
3. Извлекает ID из шапки (`process X ...` или `context X ...`).
4. Проверяет совпадение ID из имени файла и ID из шапки.

При многострочном объявлении ребёнка компилятор должен воспринимать переносы после `->` и после запятых как обычные пробелы. Диагностика по возможности указывает конкретную физическую строку, а если это невозможно — строку начала логического объявления.

## Сборка модели

После парсинга всех файлов компилятор:

1. Идентифицирует все проекты: ищет файлы `A0.*.idef0` и для каждого определяет корневую папку проекта.
2. Вычисляет имя проекта по пути от `src/idef0/` до корневой папки проекта (см. `01-dsl.md`, раздел «Имя проекта»).
3. Группирует файлы по проектам: каждый `.idef0` файл принадлежит проекту, в корневую папку которого он входит (на любой глубине).
4. Внутри каждого проекта строит иерархию декомпозиции по ID процессов (используя классическую IDEF0 нумерацию: `A11` — ребёнок `A1`, `A1A` — ребёнок `A1`).
5. Сверяет интерфейс каждого процесса в его собственном файле с тем, как этот процесс описан в строке-ребёнке у родителя.

## Правила валидации

### Правила ICOM-согласованности

1. **Покрытие интерфейса родителя.** Каждый input родителя должен быть использован хотя бы одним ребёнком. То же для outputs, controls, mechanisms. Висящие стрелки родителя — ошибка.

2. **Источник стрелок у ребёнка.** Каждая стрелка, потребляемая ребёнком, обязана либо приходить с границы листа (быть стрелкой родителя), либо приходить от другого ребёнка (как его output), либо приходить из туннеля (T-стрелка из контекстного файла A-0).

3. **Сохранение роли через границу.** Стрелки родителя сохраняют свой ICOM-тип при пересечении границы:
   - Input родителя у ребёнка — Input (не Control, не Mechanism).
   - Control родителя у ребёнка — Control.
   - Mechanism родителя у ребёнка — Mechanism.
   - Output родителя — производится одним из функциональных блоков.

4. **Согласованность интерфейса при декомпозиции.** Когда процесс имеет собственный файл декомпозиции, интерфейс в шапке этого файла должен совпадать с описанием процесса как ребёнка у родителя.

### Правила для туннелей

5. **Использование туннелей.** Каждый туннель `T*`, объявленный в `A-0.*.idef0`, должен быть использован хотя бы один раз в проекте (либо как производимая `X[T]`, либо как потребляемая `I[T]`/`C[T]`/`M[T]`). Неиспользованный туннель — ошибка.

6. **Туннель на одном листе.** На одном листе декомпозиции туннель не может быть одновременно произведён и потреблён. Если возникает такой случай — это нормальная internal-связь через X-механизм, не туннель.

### Правила именования

7. **Корень модели.** Корневой процесс проекта — всегда `A0`. Любой другой id как корень — ошибка.

8. **Правильность ID.** ID процессов следуют классической IDEF0 нумерации: корень `A0`, дети образуются суффиксами `1..9, A..Z` (заглавные). Неправильный формат ID — ошибка.

9. **Уникальность ID.** В пределах проекта каждый ID процесса встречается ровно один раз. Два файла с одним ID — ошибка коллизии.

10. **Совпадение имени файла и ID.** ID, извлечённый из имени файла, должен совпадать с ID в шапке. Несовпадение — ошибка.

### Правила структуры проекта

11. **Маркер проекта.** Корневая папка проекта определяется наличием в ней файла `A0.*.idef0`. Этот файл обязателен.

12. **Контекстный файл.** В корневой папке проекта рядом с `A0.*.idef0` обязан лежать `A-0.*.idef0`. Отсутствие — ошибка. Файл `A-0.*.idef0` в любом другом месте проекта — ошибка.

13. **Имя проекта.** Имена папок на пути от `src/idef0/` до корневой папки проекта обязаны быть валидными идентификаторами Java-пакета: первый символ — `a..z`, последующие — `a..z`, `0..9`, `_`. Точки в именах папок на этом пути запрещены. Нарушение — ошибка.

14. **Минимальная глубина.** Между `src/idef0/` и корневой папкой проекта обязан быть минимум один уровень вложенности. `src/idef0/A0.*.idef0` (A0 непосредственно в корне сканирования) — ошибка.

15. **Запрет вложенных проектов.** Внутри корневой папки проекта на любой глубине не должно быть другого `A0.*.idef0`. Вложенный проект — ошибка.

16. **Принадлежность файла проекту.** Каждый `.idef0` файл под `src/idef0/` обязан принадлежать какому-то проекту (то есть лежать внутри корневой папки какого-то проекта). Файл-сирота — ошибка.

### Расширения

17. **Количество функциональных блоков у процесса.**
    - От 0 до 9 функциональных блоков — OK.
    - От 10 до 35 функциональных блоков (с использованием суффиксов AA..AZ) — warning «декомпозиция превышает рекомендуемый предел в 9 элементов».
    - Более 35 функциональных блоков — ошибка (за пределами разрешённого алфавита).

## Виды диагностики

- **Errors** — модель не валидна, компиляция не проходит.
- **Warnings** — модель валидна, но есть отклонения от рекомендаций.
- **Info** — справочные сообщения (опционально).

## Выход

В первой версии компилятор выдаёт:

- Список ошибок и предупреждений с указанием файла, строки и сути.
- Код возврата: 0 при отсутствии ошибок, ненулевой при их наличии.

Возможное расширение в будущем — экспорт модели в промежуточный формат (JSON или подобный) для использования другими инструментами. В первой версии не требуется.

## Что компилятор не делает

- Не генерирует SVG, DOT или другие визуальные форматы.
- Не интерпретирует структуру подпапок внутри проекта (это зона плагина).
- Не валидирует семантическую корректность бизнес-смысла модели (только формальные правила нотации).
- Не рассматривает файлы вне `src/idef0/`.
