# DSL — спецификация языка

## Назначение

Текстовый формат для описания IDEF0-моделей. Один файл `.idef0` описывает **декомпозицию одной активности** — то есть на одном файле живёт один «лист» IDEF0 со своим интерфейсом и функциональными блоками.

## Базовая структура файла

```
activity <ID> "<имя активности>" {
    # объявления граничных стрелок (интерфейс активности)
    <стрелки>

    # функциональные блоки с их связями
    <функциональные блоки>
}
```

Компилятор полностью прекращает анализ файла сразу после закрывающей скобки `}`. Любой текст, заметки, фрагменты кода и черновики, расположенные после `}`, будут проигнорированы.

Принадлежность файла к проекту определяется его расположением в файловом дереве, а не объявлением в шапке (см. раздел «Проект»).

## Проект

**Проект** — связное поддерево `.idef0` файлов с единым id-пространством.

**Корневая папка проекта** — папка, содержащая файл `A0.*.idef0`. Этот файл — маркер проекта.

Структурные правила:

1. В корневой папке проекта обязательно лежат `A0.*.idef0` и `A-0.*.idef0` (см. раздел «Контекстный файл A-0»).
2. Все остальные `.idef0` файлы проекта лежат строго внутри корневой папки на произвольной глубине, и только они образуют проект.
3. Внутри корневой папки проекта на любой глубине запрещён второй `A0.*.idef0`: вложенные проекты не допускаются.
4. Структура подпапок внутри проекта — инструмент для людей, компилятор её не интерпретирует (см. `03-layout.md`).

### Имя проекта

Имя проекта вычисляется по пути от корня сканирования компилятора (`src/idef0/`) до корневой папки проекта с заменой `/` на `.`.

Имена папок на этом пути обязаны быть валидными идентификаторами в стиле Java-пакета:

- первый символ — строчная латинская буква (`a..z`);
- последующие символы — строчные латинские буквы, цифры или подчёркивание;
- точки в именах папок на этом пути запрещены (косметика недопустима).

Между `src/idef0/` и корневой папкой проекта обязан быть минимум один уровень вложенности — имя проекта не может быть пустым.

Примеры валидных имён проектов:

- `src/idef0/coffee/brewing/A0.idef0` → `coffee.brewing`
- `src/idef0/banking/lending/as_is/A0.idef0` → `banking.lending.as_is`

Невалидно:

- `src/idef0/A0.idef0` — пустое имя проекта;
- `src/idef0/Banking/lending/A0.idef0` — заглавные буквы;
- `src/idef0/banking/as-is/A0.idef0` — дефис;
- `src/idef0/банкинг/lending/A0.idef0` — не ASCII;
- `src/idef0/banking/lending.v1/A0.idef0` — точка в имени папки на пути до корня проекта.

## Идентификаторы

### Функциональные блоки

- Корневая активность модели — всегда `A0`.
- Функциональные блоки формируются конкатенацией суффиксов: `A1`, `A2`, ..., `A9`, `AA`, `AB`, ..., `AZ` (заглавные буквы латиницы).
- Внуки — конкатенация дальше: `A11`, `A12`, ..., `A1A`, `A1B`, ..., `A1Z`.
- Только заглавные `A..Z`. Строчные запрещены.
- Soft limit: до 9 функциональных блоков в одной активности (классическая рекомендация IDEF0). При 10..35 блоках компилятор выдаёт warning.
- Hard limit: 36 блоков. Больше — ошибка компиляции.

### Стрелки

Тип стрелки кодируется первой буквой её id:

- `I*` — Input (вход)
- `O*` — Output (выход)
- `C*` — Control (управление)
- `M*` — Mechanism (механизм)
- `X*` — Internal arrow (внутренняя стрелка между функциональными блоками на одном листе)
- `T*` — Tunnel (туннелированная стрелка, объявленная в контексте A-0)

Дальше может идти любая алфавитно-цифровая комбинация: `I1`, `O2`, `C42`, `X11`, `T1`.

## Граничные стрелки (граничные стрелки)

В шапке объявляются стрелки родителя — интерфейс активности:

```
activity A0 "Приготовление напитка" {
    I1 "Свежеобжаренное зерно";
    I2 "Фильтрованная вода";
    O1 "Чашка кофе";
    O2 "Отработанный жмых";
    C1 "Рецепт";
    C2 "Целевой TDS";
    M1 "Бариста";
    M2 "Кофемолка";
    ...
}
```

Каждая стрелка — объявление, которое должно заканчиваться точкой с запятой `;`. Тип определяется первой буквой id. Текст в кавычках — человекочитаемое описание.

## Декомпозиция (функциональные блоки)

Каждый функциональный блок описывается одним **логическим объявлением**, завершающимся точкой с запятой `;`:

```
A1 "Имя блока" : <потребляемые стрелки> -> <производимые стрелки>;
```

Слева от `->` — что блок принимает (входы, controls, mechanisms). Справа — что блок производит (только outputs).

Использование точки с запятой в качестве явного терминатора снимает грамматическую двусмысленность и позволяет делать переносы строк в любом месте (как в Java или C). Запятые разделяют элементы в списках. Правила и рекомендации по отступам, выравниванию в колонки и переносам длинных строк вынесены в `02-formatting.md`.

Пример логического объявления:

```
A1 "Помол зерна" : I1, C1, M1, M2 -> X11 "Смолотый кофе", X12[T1];
```

Логическое объявление функционального блока заканчивается символом `;`.

### Правила записи потребляемых стрелок

- **Граничные стрелки родителя** (`I*`, `C*`, `M*`) указываются по id без префикса роли: `I1`, `C1`, `M1`. Их роль фиксирована и совпадает с ролью у родителя.
- **Internal-стрелки от соседей** (`X*`) и **туннели** (`T*`) указываются с явным префиксом роли: `I[X11]`, `C[X11]`, `M[X11]`, `I[T1]`, `C[T1]`, `M[T1]`. Это нужно, потому что одна и та же `X`/`T` стрелка может потребляться разными блоками в разных ролях, и без префикса роль неоднозначна.

### Правила записи производимых стрелок

В правой части `->` функциональный блок производит стрелки. Возможны варианты:

- `X11 "Описание"` — создание новой internal-стрелки, которая потребляется другими функциональными блоками на этом же листе.
- `X11[O1]` — internal-стрелка, которая одновременно является **outputом родителя** O1 (уходит на границу). Имя и описание наследуются от O1.
- `X11[T1]` — internal-стрелка, которая одновременно **туннелируется** как T1 (уходит «вовне», за пределы этого листа, но не как output родителя).

**Правило:** в правой части `->` всегда используется `X`-id, даже если стрелка никем не потребляется на этом листе. Это обеспечивает консистентность и не требует рефакторинга при добавлении первого потребителя.

## Контекстный файл A-0

Каждый проект имеет файл `A-0.*.idef0`, лежащий в корневой папке проекта рядом с `A0.*.idef0`. Это **глобальный контекст модели** — контейнер для туннелированных стрелок.

```
context A-0 "Контекст использования варки кофе" {
    T1 "Шум в офисе"
    T2 "Municipal Bylaw"

    ...A0
}
```

Содержит:

- Плоский список туннелированных стрелок `T*` с описаниями.
- Опциональную ссылку `...A0` на корневая активность модели (компилятор проверяет, что A0 существует в этом же проекте).

A-0 — это **не активность** в нашей конвенции. Это контейнер уровня проекта. В классической IDEF0 A-0 — родитель A0 с единственным блоком, у нас это сознательное отступление от классики.

## Правила для туннелей

- Туннель `T*` объявляется один раз в `A-0.idef0`.
- Туннель должен быть **где-то использован** в проекте (на каком-то листе, как производимая `X[T]` или потребляемая `I[T]`/`C[T]`/`M[T]`). Иначе — ошибка компиляции (мёртвое объявление).
- На **одном листе** туннель не может быть одновременно произведён и потреблён — это уже internal-связь через X, а не туннель.
- На **одном листе** туннель может потребляться несколькими блоками в разных ролях (один блок как control, другой блок как input).
- **Глобально по проекту** туннель может играть разные роли на разных листах (например, на одном листе `C[T1]`, на другом `X22[T1]`).
- Роль не закрепляется за туннелем при объявлении (как в классической IDEF0).

## Правила согласованности интерфейса между уровнями

Когда активность `A1` имеет свой файл `A1.idef0` (то есть декомпозируется), интерфейс в шапке этого файла должен совпадать с тем, как `A1` описан в строке функционального блока у родителя `A0`:

- Те же inputs (`I*`), outputs (`O*`), controls (`C*`), mechanisms (`M*`).
- В файле декомпозируемого блока собственный нумерационный простор для его внутренних стрелок (`X*`).

## Комментарии

Однострочные: `# ...` до конца строки.

Многострочные комментарии явно внутри блока активности не поддерживаются. Однако, поскольку анализ файла прекращается после `}`, любой текст после описания активности работает как неформатированный "подвал" или многострочный комментарий.

## Строковые литералы

Только двойные кавычки. Внутренние двойные кавычки экранируются `\"`.

```
"Описание с \"цитатой\" внутри"
```

Одинарные кавычки — обычный символ, экранирование не требуется.

## Имя файла

Формат: `<ID>[.<косметика>].idef0`

- До первой точки — обязательный ID активности. Компилятор его извлекает и сверяет с id в шапке файла.
- После первой точки и до `.idef0` — необязательное человекочитаемое описание для удобства навигации.
- Косметика может содержать: буквы (любых алфавитов), цифры, подчёркивания, дефисы.
- Запрещены в косметике: пробелы, точки, слэши, кавычки.

Примеры валидных имён:

```
A0.idef0
A0.Приём_заявок.idef0
A11.Проверить-комплектность.idef0
A1.Onboarding.idef0
```

Все эквивалентны для компилятора. Косметика — для людей.

## Полный пример

```
# src/idef0/coffee/brewing/A-0.idef0

context A-0 "Контекст приготовления кофе" {
    T1 "Шум в офисе";

    ...A0;
}
```

```
# src/idef0/coffee/brewing/A0.idef0

activity A0 "Приготовление напитка" {
    I1 "Свежеобжаренное зерно";
    I2 "Фильтрованная вода";
    O1 "Чашка кофе";
    O2 "Отработанный жмых";
    C1 "Рецепт";
    C2 "Целевой TDS";
    M1 "Бариста";
    M2 "Кофемолка";
    M3 "Чайник с терморегулятором";
    M4 "Воронка и фильтр";

    A1 "Помол зерна"     : I1, C1, M1, M2 -> X11 "Смолотый кофе", X12[T1];
    A2 "Подготовка воды" : I2, C1, M3     -> X21 "Вода нужной температуры";
    A3 "Экстракция"      : I[X11], I[X21], C1, C2, M1, M4 -> X31[O1], X32[O2];
}
```
