# DSL — спецификация языка

## Назначение

Текстовый формат для описания IDEF0-моделей. Один файл `.idef0` описывает **декомпозицию одной активности** — то есть на одном файле живёт один «лист» IDEF0 со своим интерфейсом и функциональными блоками.

## Базовая структура файла

```
activity <ID> "<имя активности>" {
    # объявления граничных стрелок (boundary section)
    <стрелки>

    # функциональные блоки с их связями (decomposition section)
    <функциональные блоки>
}
```

Тело activity состоит из двух последовательных секций: сначала **boundary section** (граничные стрелки), затем **decomposition section** (функциональные блоки). Секции не перемешиваются — boundary section всегда предшествует decomposition section. Нарушение порядка — ошибка валидатора (см. `04-validator.md`, правило 18).

Парсер полностью прекращает анализ файла сразу после закрывающей скобки `}`. Любой текст, заметки, фрагменты кода и черновики, расположенные после `}`, будут проигнорированы.

Принадлежность файла к проекту определяется его расположением в файловом дереве, а не объявлением в шапке (см. раздел «Проект»).

## Границы деклараций

Декларации **не имеют явного терминатора**. Граница между декларациями определяется по **start-литералу** — первому non-whitespace токену в начале строки (после необязательного отступа).

Допустимые start-литералы зависят от контейнера и (для `activity`) от текущей секции:

| Контейнер | Секция | start-литерал | Значение |
|---|---|---|---|
| `activity` body | boundary section | `I*` / `O*` / `C*` / `M*` / `T*` | boundary arrow declaration |
| `activity` body | decomposition section | `A*` | functional block declaration |
| `context A-0` body | (одна секция) | `T*` | tunnel declaration |
| `context A-0` body | (одна секция) | `...` | root reference (`...A0`) |

В `activity` body первое появление `A*` start-литерала переключает текущую секцию с boundary на decomposition; после переключения возврат в boundary section невозможен (попытка → ошибка валидатора по правилу 18).

Декларация считается законченной перед:
- следующей строкой, начинающейся со start-литерала текущего контейнера;
- комментарием (`# ...`) в начале строки;
- закрывающей `}` тела.

Continuation lines (строки, не начинающиеся со start-литерала) — продолжение предыдущей декларации. В частности, `:`, `->`, `,`, produced-стрелки `X*`, элементы consumed-списка после переноса — всегда продолжение, потому что они не совпадают со start-литералами.

**Запятые** разделяют элементы внутри списков потребляемых и производимых стрелок. **Переносы строк** в любом месте внутри декларации допустимы и трактуются как обычный whitespace; правила и рекомендации по отступам и переносам — в `02-formatting.md`.

**Комментарии внутри декларации** запрещены — `# ...` всегда терминирует текущую декларацию. Чтобы прокомментировать блок, ставьте комментарий до или после декларации (правило прилипания — см. `02-formatting.md`).

## Проект

**Проект** — связное поддерево `.idef0` файлов с единым id-пространством.

**Корневая папка проекта** — папка, содержащая файл `A0.*.idef0`. Этот файл — маркер проекта.

Структурные правила:

1. В корневой папке проекта обязательно лежат `A0.*.idef0` и `A-0.*.idef0` (см. раздел «Контекстный файл A-0»).
2. Все остальные `.idef0` файлы проекта лежат строго внутри корневой папки на произвольной глубине, и только они образуют проект.
3. Внутри корневой папки проекта на любой глубине запрещён второй `A0.*.idef0`: вложенные проекты не допускаются.
4. Структура подпапок внутри проекта — инструмент для людей, валидатор её не интерпретирует (см. `03-layout.md`).

### Имя проекта

Имя проекта вычисляется по пути от корня сканирования валидатора (`src/idef0/`) до корневой папки проекта с заменой `/` на `.`.

Имена папок на этом пути обязаны быть валидными идентификаторами в стиле Java-пакета:

- первый символ — строчная латинская буква (`a..z`);
- последующие символы — строчные латинские буквы, цифры или подчёркивание;
- точки в именах папок на этом пути запрещены (косметика недопустима).

Между `src/idef0/` и корневой папкой проекта обязан быть минимум один уровень вложенности — имя проекта не может быть пустым.

Примеры валидных имён проектов:

- `src/idef0/coffee/brewing/A0.idef0` → `coffee.brewing`
- `src/idef0/banking/lending/as_is/A0.idef0` → `banking.lending.as_is`

Невалидно:

- `src/idef0/A0.idef0` — пустое имя проекта;
- `src/idef0/Banking/lending/A0.idef0` — заглавные буквы;
- `src/idef0/banking/as-is/A0.idef0` — дефис;
- `src/idef0/банкинг/lending/A0.idef0` — не ASCII;
- `src/idef0/banking/lending.v1/A0.idef0` — точка в имени папки на пути до корня проекта.

## Идентификаторы

### Функциональные блоки

- Корневая активность модели — всегда `A0`.
- Функциональные блоки формируются конкатенацией суффиксов: `A1`, `A2`, ..., `A9`, `Aa`, `Ab`, ..., `Az` (**строчные** буквы латиницы после префикса).
- Внуки — конкатенация дальше: `A11`, `A12`, ..., `A1a`, `A1b`, ..., `A1z`.
- Префикс `A` — **заглавный**, суффикс — только строчные `a..z` и цифры `1..9`. Заглавные буквы и `0` в суффиксе запрещены: это правило одновременно различает «префикс роли» и «путь декомпозиции» визуально, и резервирует `0` за корнем `A0`. Примеры невалидных id: `A0` в качестве дочернего (зарезервировано), `A10` (нуль внутри суффикса), `A1Z` (заглавная буква в суффиксе), `A20`, `A1z0` — все по той же причине. Формальный формат суффикса: `[1-9a-z]+`.
- Soft limit: до 9 функциональных блоков в одной активности (классическая рекомендация IDEF0). При 10..35 блоках валидатор выдаёт warning.
- Hard limit: 35 блоков (исчерпание разрешённого алфавита суффиксов `1..9, a..z` = 9 + 26 = 35). Больше — ошибка валидации.

### Стрелки

Тип стрелки кодируется первой буквой её id:

- `I*` — Input (вход)
- `O*` — Output (выход)
- `C*` — Control (управление)
- `M*` — Mechanism (механизм)
- `X*` — Internal arrow (внутренняя стрелка между функциональными блоками на одном листе)
- `T*` — Tunnel (туннелированная стрелка, объявленная в контексте A-0)

После заглавного префикса роли (`I` / `O` / `C` / `M` / `X` / `T`) идёт **строчный** алфавитно-цифровой суффикс из `a..z` и `1..9`. Цифра `0` в суффиксе запрещена — это согласуется с правилом «`0` зарезервирован» для активити и упрощает грамматику. Подчёркивания, дефисы, заглавные буквы в суффиксе, не-ASCII символы — запрещены. Формальный формат суффикса: `[1-9a-z]+`. Примеры валидных id: `I1`, `O2`, `C4`, `X11`, `T1`, `Ia`, `Cb2`, `X1a`. Невалидно: `I_1`, `i1`, `X-11`, `IA` (заглавная в суффиксе), `I10` (нуль в суффиксе).

## Plug vs Socket

В нотации IDEF0 каждая стрелка имеет источник (выход из активности) и приёмник (вход в активность). В нашем DSL эти стороны называются **plug** и **socket**:

- **Socket** — точка подключения на границе активности (boundary). Объявляется в boundary section: `I1`, `C1`, `M1`, `O1`, `T1`. Это «именованный канал», по которому стрелка пересекает границу: для входов/контролей/механизмов сокет ловит внешнюю стрелку, для выходов — выпускает внутреннюю наружу.
- **Plug** — конкретная стрелка, исходящая из дочерней активности на уровне родителя. Декларируется в produced list функционального блока: `X11`. Plug всегда принадлежит ровно одной активности (той, что его произвела) и подключается ровно к одному сокету.

Активность всегда **порождает плаг**, никогда сокет напрямую. Поэтому в produced-list декомпозиции родителя для подключения к O-сокету родителя используется compound-форма `X11[O1]`, а не просто `O1`. Аналогично для туннеля — `X11[T1]`, для own-described X — `X11[X22]`. Plug-id (`X11`) — это **обязательный** идентификатор плага на уровне декомпозиции; bracket-форма (`[O1]`/`[T1]`/`[X22]`) — обязательное указание сокета, к которому плаг подключён. Сокращённой формы `O1` в produced нет.

Соответствие plug/socket между уровнями:

- На уровне родителя: socket `O1` в boundary родителя, plug `X11[O1]` в декомпозиции — плаг X11 от блока-наследника, подключённый к сокету O1.
- При декомпозиции наследника: сокет родителя `O1` становится sokcет границы наследника — это его собственный `O1` в boundary (плоская форма). На этом уровне он уже не plug, а socket — точка, к которой подключаются плаги внуков.

**Join (несколько плагов в один сокет).** В один сокет могут подключаться несколько плагов от разных дочерних активностей (или от одной активности — это всегда join). Каждый плаг тогда получает собственный текстовый label:

```
A1 "Распознавание" : I1, C1, M1 -> X11[O1] "audio stream"
A2 "Обработка"     : I1, C1, M2 -> X21[O1] "video stream"
```

Если плаг **один** — он наследует подпись от своего сокета, label у плага запрещён:

```
A1 "Помол зерна" : I1, C1, M1, M2 -> X11[O1]
```

Подробные правила label при join — `04-validator.md`, правило 21.

## Граничные стрелки

В boundary section объявляется интерфейс активности — то, что приходит на её границу извне и то, что уходит за её границу наружу. Запись отличается для корневой активности `A0` и для декомпозированных потомков (`A1`, `A11`, …):

**В корневой активности `A0`** boundary декларирует стрелки, которые проекту дают окружение — это «вход в систему». Используются плоские формы:

```
activity A0 "Приготовление напитка" {
    I1 "Свежеобжаренное зерно"
    I2 "Фильтрованная вода"
    O1 "Чашка кофе"
    O2 "Отработанный жмых"
    C1 "Рецепт"
    C2 "Целевой TDS"
    M1 "Бариста"
    M2 "Кофемолка"
    ...
}
```

**В декомпозированных потомках** (`A1.idef0`, `A11.idef0`, …) boundary эхо-декларирует те же стрелки, которыми родитель описал блок-потомок в своей decomposition section. **Идентификатор стрелки берётся из неймспейса того уровня, где стрелка впервые появилась** (см. «Согласованность интерфейса между уровнями» ниже) — никакой локальной перенумерации.

Возможные формы в boundary:

| Форма | Откуда стрелка |
|---|---|
| `I1`, `C2`, `M3`, `O1` (плоский id) | граничная стрелка ближайшего предка (parent-I/C/M или mapped parent-O через `X[O*]`) |
| `I[X11]`, `C[X11]`, `M[X11]` | sibling-X на уровне родителя, входящий в этот блок |
| `O[X11]` | own-described `X11 "..."` родителя — этот блок производит ту X-стрелку, которую родитель назвал X11 |
| `T1` (плоский id) | туннель, упомянутый где-либо внутри этой активности (в любом её inner-блоке как `I[T1]`/`C[T1]`/`M[T1]`/`X..[T1]`). **Одна запись на туннель** независимо от того в скольких ролях и блоках он используется — роль определяется на месте использования внутри декомпозиции, а не в boundary. |

Каждая стрелка — отдельная декларация: `<form> "<описание>"`. Описание текстуально совпадает с описанием на уровне owner (см. правило description-identity в `04-validator.md`).

Пример декомпозированного потомка, родитель которого объявил `A1 ... : I1, C2, C[T1], I[X22] -> X11 "Смолотый кофе"`:

```
activity A1 "Помол зерна" {
    I1 "Свежеобжаренное зерно"
    I[X22] "Резерв"
    C2 "Целевой TDS"
    O[X11] "Смолотый кофе"
    T1 "Шум в офисе"

    A11 "..." : ...
    ...
}
```

Начало следующей строки со start-литералом (`I*`/`O*`/`C*`/`M*`/`T*` пока в boundary section, или `A*` для перехода в decomposition section) терминирует текущую декларацию. Bracket-формы вроде `I[X11]` начинаются с того же start-литерала `I` — `[X11]` это continuation на той же логической строке.

## Декомпозиция (функциональные блоки)

Каждый функциональный блок описывается одним **логическим объявлением**:

```
A1 "Имя блока" : <потребляемые стрелки> -> <производимые стрелки>
```

Слева от `->` — что блок принимает (входы, управления, механизмы). Справа — что блок производит (только выходы).

Логическое объявление может занимать несколько физических строк — переносы внутри списков трактуются как whitespace (см. `02-formatting.md` про режимы Mode 2/3 и sub-mode Б). Декларация заканчивается либо на следующей строке-`A*` (новый функциональный блок), либо на комментарии в начале строки, либо на `}`.

Пример объявления в одну строку:

```
A1 "Помол зерна" : I1, C1, M1, M2 -> X11 "Смолотый кофе", X12[T1]
```

Пример многострочного объявления (Mode 3, см. `02-formatting.md`):

```
A11 "Разбор проекта процесса"
    : I1, C1
    ->
        X111 "set process name",
        X112 "set input",
        X113 "set output"
```

Поскольку produced-стрелки начинаются с `X*`, а `X` — не start-литерал в `activity` body, парсер корректно трактует `X111`, `X112`, `X113` как продолжение декларации `A11`, а не как новые декларации.

### Правила записи потребляемых стрелок

- **Граничные стрелки родителя** (`I*`, `C*`, `M*`) указываются по id без префикса роли: `I1`, `C1`, `M1`. Их роль фиксирована и совпадает с ролью у родителя.
- **Внутренние стрелки от соседей** (`X*`) и **туннели** (`T*`) указываются с явным префиксом роли: `I[X11]`, `C[X11]`, `M[X11]`, `I[T1]`, `C[T1]`, `M[T1]`. Это нужно, потому что одна и та же `X`/`T` стрелка может потребляться разными блоками в разных ролях, и без префикса роль неоднозначна.

### Правила записи производимых стрелок

В правой части `->` функциональный блок производит стрелки. Возможны варианты:

- `X11 "Описание"` — создание **новой own-described** внутренней стрелки (sibling-X). Описание задаётся прямо здесь, эта стрелка становится «своей» для текущего уровня и может потребляться соседями через `I[X11]`/`C[X11]`/`M[X11]`. Описание-литерал в этой форме — **источник правды** для всех мест, где эта стрелка упоминается ниже по дереву.
- `X11[O1]` — плаг, подключённый к O-сокету родителя `O1` (уходит на границу текущей активности). Описание плага по умолчанию не пишется — оно наследуется от сокета (`O1 "..."` в boundary).
- `X11[T1]` — плаг, подключённый к туннелю `T1` (доступен на других листах через `I[T1]`/`C[T1]`/`M[T1]`). Описание по умолчанию не пишется — наследуется от объявления в контекстном файле (`T1 "..."`).
- `X11[X22]` — плаг, подключённый к **own-described X родителя** `X22` (этот блок-потомок «реализует» ту X-стрелку, которую родитель назвал X22 у себя в decomposition section). Описание по умолчанию не пишется — наследуется от уровня, где X22 объявлен как own-described `X22 "..."`. Эта форма нужна, когда декомпозиция блока имеет boundary `O[X22]` и какой-то inner block обязан произвести X22.

**Optional plug label при join.** Любая из трёх bracket-форм (`X11[O1]`, `X11[T1]`, `X11[X22]`) может иметь дополнительный текстовый литерал — label плага:

```
A1 "..." : I1 -> X11[O1] "audio stream"
A2 "..." : I1 -> X21[O1] "video stream"
```

Label обязателен и обязан быть указан тогда, когда в один и тот же сокет (одно и то же `[O*]`/`[T*]`/`[X*]`) на этом уровне подключается **больше одного** плага — это и есть join. Если плаг один — label запрещён (тогда подпись наследуется от сокета). Подробности — `04-validator.md`, правило 21.

**Правило:** в правой части `->` всегда используется свежий `X`-id текущего уровня, даже если стрелка никем не потребляется здесь. Это обеспечивает консистентность и не требует рефакторинга при добавлении первого потребителя. Bracket-форма `[…]` указывает, **к какому сокету** подключён плаг (parent-O, tunnel, или parent-X). Plug-id обязан использовать nomenclature `X{block_suffix}{output_index}` — см. `04-validator.md`, правило 22.

## Контекстный файл A-0

Каждый проект имеет файл `A-0.*.idef0`, лежащий в корневой папке проекта рядом с `A0.*.idef0`. Это **глобальный контекст модели** — контейнер для туннелированных стрелок.

```
context A-0 "Контекст использования варки кофе" {
    T1 "Шум в офисе"
    T2 "Municipal Bylaw"

    ...A0
}
```

Содержит:

- Плоский список туннелированных стрелок `T*` с описаниями (каждая декларация — `T<id> "<описание>"`).
- Опциональную ссылку `...A<id>` на корневую активность модели (валидатор проверяет, что указанный A-id существует в этом же проекте). Декларация ссылки начинается с `...` — три точки на старте строки служат отдельным start-литералом в context body.

A-0 — это **не активность** в нашей конвенции. Это контейнер уровня проекта. В классической IDEF0 A-0 — родитель A0 с единственным блоком, у нас это сознательное отступление от классики.

## Правила для туннелей

- Туннель `T*` объявляется один раз в `A-0.idef0`.
- Туннель должен быть **где-то использован** в проекте (на каком-то листе, как производимая `X[T]` или потребляемая `I[T]`/`C[T]`/`M[T]`). Иначе — ошибка валидации (мёртвое объявление).
- На **одном листе** туннель не может быть одновременно произведён и потреблён — это уже внутренняя связь через X, а не туннель.
- На **одном листе** туннель может потребляться несколькими блоками в разных ролях (один блок как управление, другой блок как вход).
- **Глобально по проекту** туннель может играть разные роли на разных листах (например, на одном листе `C[T1]`, на другом `X22[T1]`).
- Роль не закрепляется за туннелем при объявлении (как в классической IDEF0).
- **Туннель в boundary активности.** Если внутри декомпозиции активности `A?` любой её inner-block потребляет или производит туннель `T*` (или туннель упоминается в декомпозиции потомков `A?` на любой глубине), то `A?.boundary` обязана содержать **одну плоскую запись** `T* "<описание>"` — без префикса роли, **одна запись на туннель** независимо от того в скольких ролях и блоках он используется. Роль определяется на месте использования. Правило применяется на каждом уровне дерева декомпозиции, **включая `A0`** (т.к. туннели объявлены в `A-0` — это «внешний» по отношению к A0 уровень). Цель: читая один `.idef0` файл, видеть какие туннели он затрагивает, без открытия `A-0`.

## Согласованность интерфейса между уровнями

**Inherit-ID.** Идентификатор стрелки присваивается на том уровне, где стрелка появляется впервые, и распространяется вниз по дереву декомпозиции без изменений. Уровень введения зависит от типа стрелки:

| Тип стрелки | Где впервые объявлена | Кто owner описания |
|---|---|---|
| `I*` / `C*` / `M*` корневой активности | boundary `A0` | сам `A0` |
| Own-described X (`X22 "..."`) | decomposition section родителя в produced list | парент-блок-объявление |
| Tunnel `T*` | контекстный файл `A-0` | `A-0` |
| Mapped parent-O (`O*` через `X[O*]`) | boundary родителя как `O*` | родитель |

Когда блок `A4` имеет свой файл `A4.idef0`, его boundary эхо-объявляет ровно те же стрелки, что родитель `A0` упомянул в строке функционального блока `A4 ... : ... -> ...`:

- каждый parent-consumed `I*`/`C*`/`M*` → плоская `I*`/`C*`/`M*` в boundary `A4`,
- каждый sibling-X consumed `I[X*]`/`C[X*]`/`M[X*]` → та же bracket-форма в boundary `A4`,
- каждый produced `X[O*]` родителя → плоский `O*` в boundary `A4`,
- каждый produced own-described `X11 "..."` родителя → bracket `O[X11]` в boundary `A4`,
- каждый туннель `T*`, потреблённый родителем в `A4` (`I[T*]`/`C[T*]`/`M[T*]`) или произведённый через `A4` (`X..[T*]`), а также упомянутый где-либо ниже по декомпозиции `A4` — **одна** плоская запись `T*` в boundary `A4`, без префикса роли.

ID протаскиваются буквально, без локальной перенумерации. Это значит, что у внука `A11` в boundary может оказаться `C2` без `C1` — нумерация наследуется с верхних уровней и **не** обязана быть 1-based на каждом уровне.

Внутри самого декомпозированного блока (его decomposition section) функциональные блоки получают собственный X-неймспейс (`X11`, `X12`, …) — это локально, не наследуется от родителя. Связь с родительскими X-стрелками выражается через bracket-формы produced (`X11[X22]`) и через bracket-формы boundary (`O[X22]`).

Описание каждой стрелки на уровне-потомке обязано символ-в-символ совпадать с описанием на уровне owner — расхождение ловится валидатором (правило description-identity, см. `04-validator.md`).

## Комментарии

Однострочные: `# ...` до конца строки.

Многострочные комментарии явно внутри блока активности не поддерживаются. Однако, поскольку анализ файла прекращается после `}`, любой текст после описания активности работает как неформатированный "подвал" или многострочный комментарий.

## Строковые литералы

Только двойные кавычки. Внутренние двойные кавычки экранируются `\"`.

```
"Описание с \"цитатой\" внутри"
```

Одинарные кавычки — обычный символ, экранирование не требуется.

## Имя файла

Формат: `<ID>[.<косметика>].idef0`

- До первой точки — обязательный ID активности. Парсер его извлекает и сверяет с id в шапке файла.
- После первой точки и до `.idef0` — необязательное человекочитаемое описание для удобства навигации.
- Косметика может содержать: буквы (любых алфавитов), цифры, подчёркивания, дефисы.
- Запрещены в косметике: пробелы, точки, слэши, кавычки.

Примеры валидных имён:

```
A0.idef0
A0.Приём_заявок.idef0
A11.Проверить-комплектность.idef0
A1.Onboarding.idef0
```

Все эквивалентны для валидатора. Косметика — для людей.

## Полный пример

```
# src/idef0/coffee/brewing/A-0.idef0

context A-0 "Контекст приготовления кофе" {
    T1 "Шум в офисе"

    ...A0
}
```

```
# src/idef0/coffee/brewing/A0.idef0

activity A0 "Приготовление напитка" {
    I1 "Свежеобжаренное зерно"
    I2 "Фильтрованная вода"
    C1 "Рецепт"
    C2 "Целевой TDS"
    M1 "Бариста"
    M2 "Кофемолка"
    M3 "Чайник с терморегулятором"
    M4 "Воронка и фильтр"
    O1 "Чашка кофе"
    O2 "Отработанный жмых"
    T1 "Шум в офисе"

    A1 "Помол зерна"     : I1, C1, M1, M2 -> X11 "Смолотый кофе", X12[T1]
    A2 "Подготовка воды" : I2, C1, M3     -> X21 "Вода нужной температуры"
    A3 "Экстракция"      : I[X11], I[X21], C1, C2, M1, M4 -> X31[O1], X32[O2]
}
```

Обрати внимание — в boundary `A0` есть запись `T1 "Шум в офисе"`, хотя сам `A0` напрямую туннель не объявляет (объявление в `A-0`). Это требование правила tunnel-in-boundary: раз внутри `A0` блок `A1` производит `X12[T1]`, туннель пересекает границу `A0`, поэтому обязан быть в её boundary плоской записью.

Если `A1` декомпозируется, его файл выглядит так — boundary эхо-объявляет ровно те стрелки, которыми `A0` описал блок `A1`, и описания дублируют owner-уровень:

```
# src/idef0/coffee/brewing/A1.idef0

activity A1 "Помол зерна" {
    I1 "Свежеобжаренное зерно"
    C1 "Рецепт"
    M1 "Бариста"
    M2 "Кофемолка"
    O[X11] "Смолотый кофе"
    T1 "Шум в офисе"

    A11 "Разрушение оболочки" : I1, C1, M1 -> X111 "Дроблёное зерно"
    A12 "Финальный помол"     : I[X111], C1, M2 -> X121[X11], X122[T1]
}
```

Обрати внимание:
- В boundary `A1` нет своего номера `O1` — выход (то, что становится `X11 "Смолотый кофе"` на уровне A0) выражен bracket-формой `O[X11]`. Туннель — плоской записью `T1`, как в `A0` и в `A-0`.
- Внутри `A1` блок `A12` производит `X121[X11]` — это «мой локальный X121 это та же стрелка, что A0 у себя назвал X11». То же для `X122[T1]` — локальный X122 совпадает с туннелем T1.
- `A11` и `A12` сами не декомпозируются в этом примере, но если бы декомпозировались — их boundary тоже эхо-объявляли бы parent-arrows с теми же ID и описаниями.
