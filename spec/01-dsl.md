# DSL — спецификация языка

## Назначение

Текстовый формат для описания IDEF0-моделей. Один файл `.idef0` описывает
**декомпозицию одного процесса** — то есть на одном файле живёт один «лист»
IDEF0 со своим интерфейсом и детьми.

## Базовая структура файла

```
process <ID> "<имя процесса>" {
    # объявления граничных стрелок (интерфейс процесса)
    <стрелки>

    # дети с их связями
    <дети>
}
```

Принадлежность файла к проекту определяется его расположением в файловом
дереве, а не объявлением в шапке (см. раздел «Проект»).

## Проект

**Проект** — связное поддерево `.idef0` файлов с единым id-пространством.

**Корневая папка проекта** — папка, содержащая файл `A0.*.idef0`. Этот
файл — маркер проекта.

Структурные правила:

1. В корневой папке проекта обязательно лежат `A0.*.idef0` и `A-0.*.idef0`
   (см. раздел «Контекстный файл A-0»).
2. Все остальные `.idef0` файлы проекта лежат строго внутри корневой
   папки на произвольной глубине, и только они образуют проект.
3. Внутри корневой папки проекта на любой глубине запрещён второй
   `A0.*.idef0`: вложенные проекты не допускаются.
4. Структура подпапок внутри проекта — инструмент для людей, компилятор
   её не интерпретирует (см. `02-layout.md`).

### Имя проекта

Имя проекта вычисляется по пути от корня сканирования компилятора
(`src/idef0/`) до корневой папки проекта с заменой `/` на `.`.

Имена папок на этом пути обязаны быть валидными идентификаторами в
стиле Java-пакета:

- первый символ — строчная латинская буква (`a..z`);
- последующие символы — строчные латинские буквы, цифры или подчёркивание;
- точки в именах папок на этом пути запрещены (косметика недопустима).

Между `src/idef0/` и корневой папкой проекта обязан быть минимум один
уровень вложенности — имя проекта не может быть пустым.

Примеры валидных имён проектов:

- `src/idef0/coffee/brewing/A0.idef0` → `coffee.brewing`
- `src/idef0/banking/lending/as_is/A0.idef0` → `banking.lending.as_is`

Невалидно:

- `src/idef0/A0.idef0` — пустое имя проекта;
- `src/idef0/Banking/lending/A0.idef0` — заглавные буквы;
- `src/idef0/banking/as-is/A0.idef0` — дефис;
- `src/idef0/банкинг/lending/A0.idef0` — не ASCII;
- `src/idef0/banking/lending.v1/A0.idef0` — точка в имени папки на пути
  до корня проекта.

## Идентификаторы

### Процессы

- Корневой процесс модели — всегда `A0`.
- Дети процесса формируются конкатенацией суффиксов: `A1`, `A2`, ..., `A9`,
  `AA`, `AB`, ..., `AZ` (заглавные буквы латиницы).
- Внуки — конкатенация дальше: `A11`, `A12`, ..., `A1A`, `A1B`, ..., `A1Z`.
- Только заглавные `A..Z`. Строчные запрещены.
- Soft limit: до 9 детей у одного процесса (классическая рекомендация IDEF0).
  При 10..35 детях компилятор выдаёт warning.
- Hard limit: 36 детей. Больше — ошибка компиляции.

### Стрелки

Тип стрелки кодируется первой буквой её id:

- `I*` — Input (вход)
- `O*` — Output (выход)
- `C*` — Control (управление)
- `M*` — Mechanism (механизм)
- `X*` — Internal arrow (внутренняя стрелка между детьми на одном листе)
- `T*` — Tunnel (туннелированная стрелка, объявленная в контексте A-0)

Дальше может идти любая алфавитно-цифровая комбинация:
`I1`, `O2`, `C42`, `X11`, `T1`.

## Шапка процесса (граничные стрелки)

В шапке объявляются стрелки родителя — интерфейс процесса:

```
process A0 "Приготовление напитка" {
    I1 "Свежеобжаренное зерно"
    I2 "Фильтрованная вода"
    O1 "Чашка кофе"
    O2 "Отработанный жмых"
    C1 "Рецепт"
    C2 "Целевой TDS"
    M1 "Бариста"
    M2 "Кофемолка"
    ...
}
```

Каждая стрелка — на отдельной строке. Тип определяется первой буквой id.
Текст в кавычках — человекочитаемое описание.

## Дети процесса

Каждый ребёнок описывается одним **логическим объявлением**:

```
A1 "Имя ребёнка" : <потребляемые стрелки> -> <производимые стрелки>
```

Слева от `->` — что ребёнок принимает (входы, controls, mechanisms).
Справа — что производит (только outputs).

Короткие объявления обычно пишутся в одну физическую строку:

```
A1 "Помол зерна"     : I1, C1, M1, M2 -> X11 "Смолотый кофе"
A2 "Подготовка воды" : I2, C1, M3     -> X21 "Вода нужной температуры"
A3 "Экстракция"      : I[X11], I[X21], C1, C2, M1, M4 -> X31[O1], X32[O2]
```

Длинные объявления можно переносить на несколько физических строк. Переносы
разрешены после `->` и после запятых в списках потребляемых или производимых
стрелок. Запятая остаётся разделителем элементов; сам перенос строки элемент
не разделяет. Перенос внутри ID стрелки, внутри ID процесса или внутри строкового
литерала запрещён.

```
A1 "Помол зерна" :
    I1,
    C1,
    M1,
    M2 ->
    X11 "Смолотый кофе",
    X12[T1]

A3 "Экстракция" : I[X11], I[X21], C1, C2, M1, M4 ->
    X31[O1],
    X32[O2]
```

Логическое объявление ребёнка заканчивается перед следующим объявлением ребёнка
или перед закрывающей `}` текущего процесса.

### Правила записи потребляемых стрелок

- **Граничные стрелки родителя** (`I*`, `C*`, `M*`) указываются по id без префикса
  роли: `I1`, `C1`, `M1`. Их роль фиксирована и совпадает с ролью у родителя.
- **Internal-стрелки от соседей** (`X*`) и **туннели** (`T*`) указываются
  с явным префиксом роли: `I[X11]`, `C[X11]`, `M[X11]`, `I[T1]`, `C[T1]`, `M[T1]`.
  Это нужно, потому что одна и та же `X`/`T` стрелка может потребляться разными
  детьми в разных ролях, и без префикса роль неоднозначна.

### Правила записи производимых стрелок

В правой части `->` ребёнок производит стрелки. Возможны варианты:

- `X11 "Описание"` — создание новой internal-стрелки, которая
  потребляется другими детьми на этом же листе.
- `X11[O1]` — internal-стрелка, которая одновременно является **outputом
  родителя** O1 (уходит на границу). Имя и описание наследуются от O1.
- `X11[T1]` — internal-стрелка, которая одновременно **туннелируется** как T1
  (уходит «вовне», за пределы этого листа, но не как output родителя).

**Правило:** в правой части `->` всегда используется `X`-id, даже если стрелка
никем не потребляется на этом листе. Это обеспечивает консистентность и
не требует рефакторинга при добавлении первого потребителя.

## Контекстный файл A-0

Каждый проект имеет файл `A-0.*.idef0`, лежащий в корневой папке проекта
рядом с `A0.*.idef0`. Это **глобальный контекст модели** — контейнер
для туннелированных стрелок.

```
context A-0 "Контекст использования варки кофе" {
    T1 "Шум в офисе"
    T2 "Municipal Bylaw"

    ...A0
}
```

Содержит:

- Плоский список туннелированных стрелок `T*` с описаниями.
- Опциональную ссылку `...A0` на корневой процесс модели (компилятор
  проверяет, что A0 существует в этом же проекте).

A-0 — это **не процесс** в нашей конвенции. Это контейнер уровня проекта.
В классической IDEF0 A-0 — родитель A0 с единственным блоком, у нас это
сознательное отступление от классики.

## Правила для туннелей

- Туннель `T*` объявляется один раз в `A-0.idef0`.
- Туннель должен быть **где-то использован** в проекте (на каком-то листе,
  как производимая `X[T]` или потребляемая `I[T]`/`C[T]`/`M[T]`). Иначе —
  ошибка компиляции (мёртвое объявление).
- На **одном листе** туннель не может быть одновременно произведён и
  потреблён — это уже internal-связь через X, а не туннель.
- На **одном листе** туннель может потребляться несколькими детьми в разных
  ролях (один ребёнок как control, другой как input).
- **Глобально по проекту** туннель может играть разные роли на разных листах
  (например, на одном листе `C[T1]`, на другом `X22[T1]`).
- Роль не закрепляется за туннелем при объявлении (как в классической IDEF0).

## Правила согласованности интерфейса между уровнями

Когда процесс `A1` имеет свой файл `A1.idef0` (то есть декомпозируется),
интерфейс в шапке этого файла должен совпадать с тем, как `A1` описан
в строке-ребёнке у родителя `A0`:

- Те же inputs (`I*`), outputs (`O*`), controls (`C*`), mechanisms (`M*`).
- В файле ребёнка собственный нумерационный простор для его внутренних
  стрелок (`X*`).

## Комментарии

Однострочные: `# ...` до конца строки.

Многострочные комментарии не поддерживаются.

## Строковые литералы

Только двойные кавычки. Внутренние двойные кавычки экранируются `\"`.

```
"Описание с \"цитатой\" внутри"
```

Одинарные кавычки — обычный символ, экранирование не требуется.

## Имя файла

Формат: `<ID>[.<косметика>].idef0`

- До первой точки — обязательный ID процесса. Компилятор его извлекает
  и сверяет с id в шапке файла.
- После первой точки и до `.idef0` — необязательное человекочитаемое
  описание для удобства навигации.
- Косметика может содержать: буквы (любых алфавитов), цифры, подчёркивания,
  дефисы.
- Запрещены в косметике: пробелы, точки, слэши, кавычки.

Примеры валидных имён:

```
A0.idef0
A0.Приём_заявок.idef0
A11.Проверить-комплектность.idef0
A1.Onboarding.idef0
```

Все эквивалентны для компилятора. Косметика — для людей.

## Полный пример

```
# src/idef0/coffee/brewing/A-0.idef0

context A-0 "Контекст приготовления кофе" {
    T1 "Шум в офисе"

    ...A0
}
```

```
# src/idef0/coffee/brewing/A0.idef0

process A0 "Приготовление напитка" {
    I1 "Свежеобжаренное зерно"
    I2 "Фильтрованная вода"
    O1 "Чашка кофе"
    O2 "Отработанный жмых"
    C1 "Рецепт"
    C2 "Целевой TDS"
    M1 "Бариста"
    M2 "Кофемолка"
    M3 "Чайник с терморегулятором"
    M4 "Воронка и фильтр"

    A1 "Помол зерна"     : I1, C1, M1, M2 -> X11 "Смолотый кофе", X12[T1]
    A2 "Подготовка воды" : I2, C1, M3     -> X21 "Вода нужной температуры"
    A3 "Экстракция"      : I[X11], I[X21], C1, C2, M1, M4 -> X31[O1], X32[O2]
}
```
