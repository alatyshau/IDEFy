# DSL — спецификация языка

## Назначение

Текстовый формат для описания IDEF0-моделей. Один файл `.idef0` описывает **декомпозицию одной активности** — то есть на одном файле живёт один «лист» IDEF0 со своим интерфейсом и функциональными блоками.

## Базовая структура файла

```
activity <ID> "<имя активности>" {
    # объявления граничных стрелок (boundary section)
    <стрелки>

    # функциональные блоки с их связями (decomposition section)
    <функциональные блоки>
}
```

Тело activity состоит из двух последовательных секций: сначала **boundary section** (граничные стрелки), затем **decomposition section** (функциональные блоки). Секции не перемешиваются — boundary section всегда предшествует decomposition section. Нарушение порядка — ошибка валидатора (см. `04-validator.md`, правило 18).

Парсер полностью прекращает анализ файла сразу после закрывающей скобки `}`. Любой текст, заметки, фрагменты кода и черновики, расположенные после `}`, будут проигнорированы.

Принадлежность файла к проекту определяется его расположением в файловом дереве, а не объявлением в шапке (см. раздел «Проект»).

## Границы деклараций

Декларации **не имеют явного терминатора**. Граница между декларациями определяется по **start-литералу** — первому non-whitespace токену в начале строки (после необязательного отступа).

Допустимые start-литералы зависят от контейнера и (для `activity`) от текущей секции:

| Контейнер | Секция | start-литерал | Значение |
|---|---|---|---|
| `activity` body | boundary section | `I*` / `O*` / `C*` / `M*` | boundary arrow declaration |
| `activity` body | decomposition section | `A*` | functional block declaration |
| `context A-0` body | (одна секция) | `T*` | tunnel declaration |
| `context A-0` body | (одна секция) | `...` | root reference (`...A0`) |

В `activity` body первое появление `A*` start-литерала переключает текущую секцию с boundary на decomposition; после переключения возврат в boundary section невозможен (попытка → ошибка валидатора по правилу 18).

Декларация считается законченной перед:
- следующей строкой, начинающейся со start-литерала текущего контейнера;
- комментарием (`# ...`) в начале строки;
- закрывающей `}` тела.

Continuation lines (строки, не начинающиеся со start-литерала) — продолжение предыдущей декларации. В частности, `:`, `->`, `,`, produced-стрелки `X*`, элементы consumed-списка после переноса — всегда продолжение, потому что они не совпадают со start-литералами.

**Запятые** разделяют элементы внутри списков потребляемых и производимых стрелок. **Переносы строк** в любом месте внутри декларации допустимы и трактуются как обычный whitespace; правила и рекомендации по отступам и переносам — в `02-formatting.md`.

**Комментарии внутри декларации** запрещены — `# ...` всегда терминирует текущую декларацию. Чтобы прокомментировать блок, ставьте комментарий до или после декларации (правило прилипания — см. `02-formatting.md`).

## Проект

**Проект** — связное поддерево `.idef0` файлов с единым id-пространством.

**Корневая папка проекта** — папка, содержащая файл `A0.*.idef0`. Этот файл — маркер проекта.

Структурные правила:

1. В корневой папке проекта обязательно лежат `A0.*.idef0` и `A-0.*.idef0` (см. раздел «Контекстный файл A-0»).
2. Все остальные `.idef0` файлы проекта лежат строго внутри корневой папки на произвольной глубине, и только они образуют проект.
3. Внутри корневой папки проекта на любой глубине запрещён второй `A0.*.idef0`: вложенные проекты не допускаются.
4. Структура подпапок внутри проекта — инструмент для людей, валидатор её не интерпретирует (см. `03-layout.md`).

### Имя проекта

Имя проекта вычисляется по пути от корня сканирования валидатора (`src/idef0/`) до корневой папки проекта с заменой `/` на `.`.

Имена папок на этом пути обязаны быть валидными идентификаторами в стиле Java-пакета:

- первый символ — строчная латинская буква (`a..z`);
- последующие символы — строчные латинские буквы, цифры или подчёркивание;
- точки в именах папок на этом пути запрещены (косметика недопустима).

Между `src/idef0/` и корневой папкой проекта обязан быть минимум один уровень вложенности — имя проекта не может быть пустым.

Примеры валидных имён проектов:

- `src/idef0/coffee/brewing/A0.idef0` → `coffee.brewing`
- `src/idef0/banking/lending/as_is/A0.idef0` → `banking.lending.as_is`

Невалидно:

- `src/idef0/A0.idef0` — пустое имя проекта;
- `src/idef0/Banking/lending/A0.idef0` — заглавные буквы;
- `src/idef0/banking/as-is/A0.idef0` — дефис;
- `src/idef0/банкинг/lending/A0.idef0` — не ASCII;
- `src/idef0/banking/lending.v1/A0.idef0` — точка в имени папки на пути до корня проекта.

## Идентификаторы

### Функциональные блоки

- Корневая активность модели — всегда `A0`.
- Функциональные блоки формируются конкатенацией суффиксов: `A1`, `A2`, ..., `A9`, `AA`, `AB`, ..., `AZ` (заглавные буквы латиницы).
- Внуки — конкатенация дальше: `A11`, `A12`, ..., `A1A`, `A1B`, ..., `A1Z`.
- Только заглавные `A..Z`. Строчные запрещены.
- Цифра `0` в суффиксе запрещена — она зарезервирована за корнем `A0`. Поэтому `A10`, `A20`, `A1Z0` — невалидные id (формат суффикса: `[1-9A-Z]+`).
- Soft limit: до 9 функциональных блоков в одной активности (классическая рекомендация IDEF0). При 10..35 блоках валидатор выдаёт warning.
- Hard limit: 35 блоков (исчерпание разрешённого алфавита суффиксов `1..9, A..Z` = 9 + 26 = 35). Больше — ошибка валидации.

### Стрелки

Тип стрелки кодируется первой буквой её id:

- `I*` — Input (вход)
- `O*` — Output (выход)
- `C*` — Control (управление)
- `M*` — Mechanism (механизм)
- `X*` — Internal arrow (внутренняя стрелка между функциональными блоками на одном листе)
- `T*` — Tunnel (туннелированная стрелка, объявленная в контексте A-0)

После префикса роли идёт алфавитно-цифровая комбинация из **заглавных латинских букв и цифр** (`A..Z` и `0..9`). Подчёркивания, дефисы, строчные буквы, не-ASCII символы в id стрелок запрещены. Примеры валидных id: `I1`, `O2`, `C42`, `X11`, `T1`. Невалидно: `I_1`, `i1`, `X-11`.

## Граничные стрелки

В boundary section объявляются стрелки родителя — интерфейс активности:

```
activity A0 "Приготовление напитка" {
    I1 "Свежеобжаренное зерно"
    I2 "Фильтрованная вода"
    O1 "Чашка кофе"
    O2 "Отработанный жмых"
    C1 "Рецепт"
    C2 "Целевой TDS"
    M1 "Бариста"
    M2 "Кофемолка"
    ...
}
```

Каждая стрелка — отдельная декларация: `<id> "<описание>"`. Тип определяется первой буквой id (см. «Идентификаторы → Стрелки»). Текст в кавычках — человекочитаемое описание. Начало следующей строки со start-литералом (`I*`/`O*`/`C*`/`M*` пока в boundary section, или `A*` для перехода в decomposition section) терминирует текущую декларацию.

## Декомпозиция (функциональные блоки)

Каждый функциональный блок описывается одним **логическим объявлением**:

```
A1 "Имя блока" : <потребляемые стрелки> -> <производимые стрелки>
```

Слева от `->` — что блок принимает (входы, управления, механизмы). Справа — что блок производит (только выходы).

Логическое объявление может занимать несколько физических строк — переносы внутри списков трактуются как whitespace (см. `02-formatting.md` про режимы Mode 2/3 и sub-mode Б). Декларация заканчивается либо на следующей строке-`A*` (новый функциональный блок), либо на комментарии в начале строки, либо на `}`.

Пример объявления в одну строку:

```
A1 "Помол зерна" : I1, C1, M1, M2 -> X11 "Смолотый кофе", X12[T1]
```

Пример многострочного объявления (Mode 3, см. `02-formatting.md`):

```
A11 "Разбор проекта процесса"
    : I1, C1
    ->
        X111 "set process name",
        X112 "set input",
        X113 "set output"
```

Поскольку produced-стрелки начинаются с `X*`, а `X` — не start-литерал в `activity` body, парсер корректно трактует `X111`, `X112`, `X113` как продолжение декларации `A11`, а не как новые декларации.

### Правила записи потребляемых стрелок

- **Граничные стрелки родителя** (`I*`, `C*`, `M*`) указываются по id без префикса роли: `I1`, `C1`, `M1`. Их роль фиксирована и совпадает с ролью у родителя.
- **Внутренние стрелки от соседей** (`X*`) и **туннели** (`T*`) указываются с явным префиксом роли: `I[X11]`, `C[X11]`, `M[X11]`, `I[T1]`, `C[T1]`, `M[T1]`. Это нужно, потому что одна и та же `X`/`T` стрелка может потребляться разными блоками в разных ролях, и без префикса роль неоднозначна.

### Правила записи производимых стрелок

В правой части `->` функциональный блок производит стрелки. Возможны варианты:

- `X11 "Описание"` — создание новой внутренней стрелки, которая потребляется другими функциональными блоками на этом же листе.
- `X11[O1]` — внутренняя стрелка, которая одновременно является **выходом родителя** O1 (уходит на границу). Имя и описание наследуются от O1.
- `X11[T1]` — внутренняя стрелка, которая одновременно **туннелируется** как T1 (уходит «вовне», за пределы этого листа, но не как выход родителя).

**Правило:** в правой части `->` всегда используется `X`-id, даже если стрелка никем не потребляется на этом листе. Это обеспечивает консистентность и не требует рефакторинга при добавлении первого потребителя.

## Контекстный файл A-0

Каждый проект имеет файл `A-0.*.idef0`, лежащий в корневой папке проекта рядом с `A0.*.idef0`. Это **глобальный контекст модели** — контейнер для туннелированных стрелок.

```
context A-0 "Контекст использования варки кофе" {
    T1 "Шум в офисе"
    T2 "Municipal Bylaw"

    ...A0
}
```

Содержит:

- Плоский список туннелированных стрелок `T*` с описаниями (каждая декларация — `T<id> "<описание>"`).
- Опциональную ссылку `...A<id>` на корневую активность модели (валидатор проверяет, что указанный A-id существует в этом же проекте). Декларация ссылки начинается с `...` — три точки на старте строки служат отдельным start-литералом в context body.

A-0 — это **не активность** в нашей конвенции. Это контейнер уровня проекта. В классической IDEF0 A-0 — родитель A0 с единственным блоком, у нас это сознательное отступление от классики.

## Правила для туннелей

- Туннель `T*` объявляется один раз в `A-0.idef0`.
- Туннель должен быть **где-то использован** в проекте (на каком-то листе, как производимая `X[T]` или потребляемая `I[T]`/`C[T]`/`M[T]`). Иначе — ошибка валидации (мёртвое объявление).
- На **одном листе** туннель не может быть одновременно произведён и потреблён — это уже внутренняя связь через X, а не туннель.
- На **одном листе** туннель может потребляться несколькими блоками в разных ролях (один блок как управление, другой блок как вход).
- **Глобально по проекту** туннель может играть разные роли на разных листах (например, на одном листе `C[T1]`, на другом `X22[T1]`).
- Роль не закрепляется за туннелем при объявлении (как в классической IDEF0).

## Правила согласованности интерфейса между уровнями

Когда активность `A1` имеет свой файл `A1.idef0` (то есть декомпозируется), интерфейс в шапке этого файла должен совпадать с тем, как `A1` описан в строке функционального блока у родителя `A0`:

- Те же inputs (`I*`), outputs (`O*`), controls (`C*`), mechanisms (`M*`).
- В файле декомпозируемого блока собственный нумерационный простор для его внутренних стрелок (`X*`).

## Комментарии

Однострочные: `# ...` до конца строки.

Многострочные комментарии явно внутри блока активности не поддерживаются. Однако, поскольку анализ файла прекращается после `}`, любой текст после описания активности работает как неформатированный "подвал" или многострочный комментарий.

## Строковые литералы

Только двойные кавычки. Внутренние двойные кавычки экранируются `\"`.

```
"Описание с \"цитатой\" внутри"
```

Одинарные кавычки — обычный символ, экранирование не требуется.

## Имя файла

Формат: `<ID>[.<косметика>].idef0`

- До первой точки — обязательный ID активности. Парсер его извлекает и сверяет с id в шапке файла.
- После первой точки и до `.idef0` — необязательное человекочитаемое описание для удобства навигации.
- Косметика может содержать: буквы (любых алфавитов), цифры, подчёркивания, дефисы.
- Запрещены в косметике: пробелы, точки, слэши, кавычки.

Примеры валидных имён:

```
A0.idef0
A0.Приём_заявок.idef0
A11.Проверить-комплектность.idef0
A1.Onboarding.idef0
```

Все эквивалентны для валидатора. Косметика — для людей.

## Полный пример

```
# src/idef0/coffee/brewing/A-0.idef0

context A-0 "Контекст приготовления кофе" {
    T1 "Шум в офисе"

    ...A0
}
```

```
# src/idef0/coffee/brewing/A0.idef0

activity A0 "Приготовление напитка" {
    I1 "Свежеобжаренное зерно"
    I2 "Фильтрованная вода"
    O1 "Чашка кофе"
    O2 "Отработанный жмых"
    C1 "Рецепт"
    C2 "Целевой TDS"
    M1 "Бариста"
    M2 "Кофемолка"
    M3 "Чайник с терморегулятором"
    M4 "Воронка и фильтр"

    A1 "Помол зерна"     : I1, C1, M1, M2 -> X11 "Смолотый кофе", X12[T1]
    A2 "Подготовка воды" : I2, C1, M3     -> X21 "Вода нужной температуры"
    A3 "Экстракция"      : I[X11], I[X21], C1, C2, M1, M4 -> X31[O1], X32[O2]
}
```
