# IDEFy — продуктовая спека

## Что есть IDEFy

IDEFy — это **текстовый DSL для моделирования бизнес-процессов в нотации IDEF0**, оптимизированный под хранение в git и редактирование в обычном текстовом редакторе. Модель читается непосредственно из исходников без рендеринга в графику.

Это **главный продукт**. Всё остальное — инструментальная обвязка, которая делает работу с DSL удобной, но не определяет его суть. DSL остаётся валидным и используемым даже без какого-либо тулинга — это просто `.idef0` текстовые файлы в git-репозитории.

Вокруг DSL построен набор инструментов:

- **Валидатор** — проверяет `.idef0` файлы на соответствие правилам нотации, возвращает диагностики.
- **Форматтер** — приводит `.idef0` к каноническому виду по строгим правилам.
- **Рендерер** — производит содержимое ASCII-схемы листа (заглушка в текущей итерации). Запись sidecar-файла `<name>.idef0.ascii` на диск — задача editor-host'а, не самого рендерера.
- **VS Code расширение** — editor-host: подсветка синтаксиса с семантической раскраской ролей стрелок, live-диагностики в стиле «red squiggles» + Problems panel, форматтер по команде, запись sidecar при формате, ASCII-схема в соседнем табе через custom editor.

Валидатор, форматтер и рендерер написаны как чистая библиотека (`@idefy/core`) без зависимостей от VS Code API и без файловых I/O — это открывает дорогу к CLI, LSP-серверу и кросс-редакторной поддержке без переписывания. FS-операции (включая запись sidecar) делает editor-host через адаптер `@idefy/loader`.

## Архитектура

Три слоя:

1. **DSL.** Текстовый формат `.idef0` файлов. Полная спецификация — [`spec/01-dsl.md`](01-dsl.md). Раскладка файлов проекта по папкам — [`spec/03-layout.md`](03-layout.md). Граница проекта определяется расположением маркерного файла `A0.*.idef0`. **Это сам продукт** — DSL валиден и используем как stand-alone артефакт в git.
2. **Валидатор, форматтер, рендерер.** Парсят `.idef0` файлы, собирают модель, валидируют по правилам [`spec/04-validator.md`](04-validator.md), форматируют по правилам [`spec/02-formatting.md`](02-formatting.md). Рендерер возвращает **содержимое** ASCII-схемы листа (строку), не пишет файл. Реализуется в пакете `@idefy/core` без зависимостей от VS Code API и без файловых I/O — это библиотека, переиспользуемая в любом editor-host'е или CLI.
3. **VS Code-расширение.** UX-обвязка вокруг валидатора и форматтера. Регистрирует языки `idef0` и `idef0.ascii`, прокидывает диагностики валидатора, регистрирует команду форматтера, добавляет в editor toolbar кнопку открытия ASCII-схемы в соседнем табе через custom editor. Реализуется в пакете `@idefy/vscode`. **Не единственный** возможный editor-host — VS Code расширение в текущей итерации, в будущем — LSP-сервер для других редакторов.

## Дистрибуция и аудитория

- Внутренний инструмент автора. Репозиторий публичный — пользуется кто хочет на свой страх и риск.
- **DSL** распространяется как код в git-репозитории: пользователь создаёт `.idef0` файлы в своём проекте, коммитит, делится через стандартные git-механизмы. Никакого отдельного «дистрибутива DSL» не существует — DSL это спецификация, а не пакет.
- **Тулинг** (VS Code расширение): локальная сборка `.vsix`, без публикации в Marketplace. Пользователь клонирует репозиторий IDEFy, собирает `.vsix`, устанавливает в свой VS Code.
