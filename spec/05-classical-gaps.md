# Classical Gaps — фичи классической IDEF0, отсутствующие в нашем DSL

Этот документ перечисляет элементы стандарта FIPS PUB 183 (IDEF0), которые **существуют в классической нотации**, но мы осознанно не реализуем в нашем DSL.

Цель документа — зафиксировать осведомлённость, чтобы через какое-то время никто не пытался изобретать решения заново, не зная, что они уже обсуждены и сознательно отложены.

Если в будущем возникнет реальная потребность в какой-то из этих фич — мы вернёмся к её обсуждению. До этого момента — out of scope.

---

## 1. Метки сегментов стрелки при ветвлении

**Что это в классике.** Когда одна стрелка ветвится на несколько направлений (например, control «Регламент» идёт ко всем детям), каждой ветке можно присвоить свою метку: «Регламент: часть про сроки», «Регламент: часть про подписи».

**Почему не сейчас.** Семантически это редкий случай. У нас стрелка одна и та же на всех ветках, без дифференциации. Если возникнет реальный кейс — добавим расширенный синтаксис.

---

## 2. Bundling / Unbundling стрелок

**Что это в классике.** Несколько стрелок логически объединяются в одну на верхнем уровне для упрощения и «разворачиваются» обратно при декомпозиции. Например, «Сырьё» на уровне A0 при детализации в A1 разворачивается в «Зерно», «Воду», «Молоко».

**Почему не сейчас.** Это оптимизация визуального восприятия больших моделей. В нашем текстовом DSL читаемость не страдает от того, что стрелки везде самостоятельны. Реализация требует механизма «составных» стрелок, который существенно усложнит нотацию.

---

## 3. Call arrows между моделями

**Что это в классике.** Стрелка-вызов, ссылающаяся на процесс в другой модели или другом проекте. Используется для интеграции независимых моделей.

**Почему не сейчас.** У нас на старте одна модель в проекте. Когда появится вторая, с которой нужна связь — обсудим. Скорее всего, решение будет через явные ссылки в DSL с указанием имени проекта.

---

## 4. Метаданные диаграмм

**Что это в классике.** Каждая диаграмма имеет заголовок с полями: автор, дата создания, статус, C-number (chronological version).

**Почему не сейчас.** Все эти метаданные предоставляет git: автор, дата, история изменений, версии. Дублировать в самом файле — лишняя работа и риск рассинхрона.

---

## 5. FEO-диаграммы (For Exposition Only)

**Что это в классике.** Иллюстративные диаграммы, не входящие в формальную модель. Используются для презентаций, объяснений, обучения.

**Почему не сейчас.** Презентации и работа с внешней аудиторией — out of scope (модели читаются только командой). Если когда-то потребуется генерировать визуалы для презентаций — это будет задача рендера, не DSL.

---

## 6. Расширенный глоссарий

**Что это в классике.** Отдельный документ модели, содержащий длинные формальные определения каждой стрелки и каждого процесса с примерами использования.

**Почему не сейчас.** У нас уже есть короткое описание в кавычках при объявлении каждой сущности. Для маленькой команды этого достаточно. Расширенный глоссарий — это требование больших проектов с многими участниками, у нас этого нет.

---

## 7. Node tree / Node index

**Что это в классике.** Автоматически генерируемое иерархическое оглавление всех узлов модели для навигации.

**Почему не сейчас.** Текстовая структура файлов и подсветка в VS Code заменяют эту функцию. При необходимости валидатор может тривиально сгенерировать такое оглавление.

---

## 8. Multiple sheets для одной диаграммы

**Что это в классике.** В бумажной IDEF0 при большой декомпозиции диаграмма разносится на несколько физических листов с переходными ссылками.

**Почему не сейчас.** Это полностью артефакт бумажного формата. В текстовом DSL длина файла не ограничена, и при большой декомпозиции файл просто длиннее обычного.

---

## 9. Approval workflow

**Что это в классике.** Формальный процесс утверждения модели разными ролями: author (создатель), commentator (рецензент), reader (читатель), librarian (хранитель версий).

**Почему не сейчас.** У нас всё через git: PR, code review, права на ветки. Дополнительный workflow в DSL избыточен.

---

## 10. DRE — Detail Reference Expressions

**Что это в классике.** Ссылочные выражения на стрелках для указания перекрёстных связей между диаграммами разных уровней детализации.

**Почему не сейчас.** Очень редко используемая фича стандарта. У нас нет такого кейса в обозримом будущем.

---

## Принцип отложенных фич

По каждой из этих фич применяется одинаковое правило: **возвращаемся к обсуждению только при появлении конкретной потребности.** Преждевременная реализация классических механизмов «на всякий случай» противоречит общему принципу YAGNI, заявленному в `00-README.md`.
