# Валидатор — спецификация поведения

## Назначение

Валидатор читает `.idef0` файлы проекта, собирает из них единую модель и проверяет её на соответствие правилам нотации. Выдаёт ошибки и предупреждения.

## Входы

Валидатор сканирует исключительно каталог `src/idef0/` рекурсивно и собирает все файлы `*.idef0` независимо от того, в какой подпапке они лежат. Файлы вне `src/idef0/` валидатору недоступны и в модель не попадают.

## Парсинг

Для каждого файла:

1. Извлекает ID активности из имени файла (часть до первой точки).
2. Разбирает содержимое с учётом логических объявлений (см. `01-dsl.md`, раздел «Границы деклараций»): декларации не имеют явного терминатора, начало новой декларации определяется по start-литералу текущей секции в начале строки.
3. Извлекает ID из шапки (`activity X ...` или `context X ...`).
4. Проверяет совпадение ID из имени файла и ID из шапки.

При многострочном объявлении функционального блока парсер должен воспринимать переносы внутри списков (после `->`, после запятых, после `:`) как обычные пробелы. Диагностика по возможности указывает конкретную физическую строку, а если это невозможно — строку начала логического объявления.

## Сборка модели

После парсинга всех файлов валидатор:

1. Идентифицирует все проекты: ищет файлы `A0.*.idef0` и для каждого определяет корневую папку проекта.
2. Вычисляет имя проекта по пути от `src/idef0/` до корневой папки проекта (см. `01-dsl.md`, раздел «Имя проекта»).
3. Группирует файлы по проектам: каждый `.idef0` файл принадлежит проекту, в корневую папку которого он входит (на любой глубине).
4. Внутри каждого проекта строит иерархию декомпозиции по ID активностей (используя IDEFy-нумерацию по `spec/01-dsl.md`: `A11` — функциональный блок-наследник `A1`, `A1a` — функциональный блок-наследник `A1`).
5. Сверяет интерфейс каждой активности в её собственном файле с тем, как эта активность описана в строке соответствующего функционального блока у родителя.

## Owner-таблица описаний (load-bearing для правил 4 и 19)

Каждый литерал-описание в DSL имеет **owner-уровень** — место, где это описание считается источником правды. Все echo-вхождения того же ID на других уровнях обязаны описанием совпадать с owner'ом символ-в-символ.

| Форма литерала | Где владелец описания |
|---|---|
| `I*`/`C*`/`M*` в boundary `A0` | сама эта строка boundary `A0` |
| `O*` в boundary `A0` | сама эта строка boundary `A0` — описание socket'а |
| `I*`/`C*`/`M*` в boundary декомпозированного потомка | плоский литерал с тем же ID в boundary ближайшего предка |
| `O*` в boundary декомпозированного потомка `A?` | label плага у предка (`X[O*] "..."`), если он указан; иначе — плоский литерал `O*` в boundary ближайшего предка (описание socket'а) |
| `O[X*]` в boundary декомпозированного потомка `A?` | label плага у предка (`X[X*] "..."`), если он указан; иначе — литерал `X* "..."` в produced list ближайшего предка (own-described) |
| `I[X*]`/`C[X*]`/`M[X*]` в boundary | литерал `X* "..."` в produced list ближайшего предка, где он впервые объявлен как own-described |
| `T*` в boundary активности `A?` | label плага у предка (`X[T*] "..."`), если он указан и этот предок производит данный туннель; иначе — литерал `T* "..."` в `A-0.context.idef0` |
| `T*` в `A-0.context.idef0` | сама эта строка — это первичное объявление |
| `X* "..."` (own-described) в produced list | сам этот литерал — это первичное объявление |
| `X[O*] "..."` / `X[T*] "..."` / `X[X*] "..."` (plug label при join) | сам этот литерал — это первичное объявление; используется как owner для соответствующего entry в boundary ребёнка |
| `activity A? "name"` header | литерал `A? "name"` в декларации функционального блока у ближайшего предка |
| `context A-0 "name"` header | сам этот литерал — это первичное объявление |

При **single plug** (один плаг подключён к сокету) label у плага запрещён (правило 21), и описание для соответствующего entry в boundary ребёнка наследуется от socket'а — это обычный case до введения join.

При **join** (≥2 плагов в один сокет) каждый плаг обязан иметь свой label (правило 21). Тогда label плага становится owner'ом описания для boundary entry в файле декомпозиции той активности, чей плаг это (один и тот же сокет, но разные label у разных потомков).

## Правила валидации

### Правила ICOM-согласованности

1. **Покрытие интерфейса родителя.** [TODO] Каждый вход родителя должен быть использован хотя бы одним функциональным блоком-наследником. То же для выходов, управлений, механизмов. Висящие стрелки родителя — ошибка.

2. **Источник стрелок у наследника.** [TODO] Каждая стрелка, потребляемая функциональным блоком, обязана либо приходить с границы листа (быть стрелкой родителя), либо приходить от другого блока на этом листе (как его выход), либо приходить из туннеля (T-стрелка из контекстного файла A-0).

3. **Сохранение роли через границу.** [TODO] Стрелки родителя сохраняют свой ICOM-тип при пересечении границы:
   - Вход родителя у наследника — используется как вход (не управление, не механизм).
   - Управление родителя у наследника — управление.
   - Механизм родителя у наследника — механизм.
   - Выход родителя — производится одним из функциональных блоков.

4. **Согласованность интерфейса при декомпозиции (inherit-ID).** Когда активность `A?` имеет собственный файл декомпозиции, её boundary section обязана **точно** соответствовать тому, что родитель (ближайший предок) объявил в своей строке функционального блока `A? ... : <consumed> -> <produced>`. Соответствие — per-ID, для каждой формы свой эквивалент в boundary потомка:

   | Что у родителя | Что в boundary потомка |
   |---|---|
   | parent-consumed flat `I*`/`C*`/`M*` | плоский `I*`/`C*`/`M*` с тем же ID |
   | parent-consumed bracket `I[X*]`/`C[X*]`/`M[X*]` | bracket `I[X*]`/`C[X*]`/`M[X*]` с тем же sibling-X ID |
   | parent-produced `X[O*]` | плоский `O*` с тем же ID |
   | parent-produced own-described `X11 "..."` | bracket `O[X11]` с тем же X-ID |
   | parent-consumed/produced туннель в любой форме | плоский `T*` (см. правило 20 tunnel-in-boundary) |

   Лишний или недостающий ID в boundary потомка относительно того, что декларировал родитель, — error. Lookups делаются **по ID**, не по описанию (description-identity — отдельное правило 19).

   Cardinality-проверка для own-described X (которая была в предыдущей итерации) **больше не нужна** — с inherit-ID у каждого own-described `X11 "..."` родителя есть конкретный counterpart `O[X11]` в boundary потомка, проверка идёт по ID точно.

### Правила для туннелей [TODO]

5. **Использование туннелей.** Каждый туннель `T*`, объявленный в `A-0.*.idef0`, должен быть использован хотя бы один раз в проекте (либо как производимая `X[T]`, либо как потребляемая `I[T]`/`C[T]`/`M[T]`). Неиспользованный туннель — ошибка.

6. **Туннель на одном листе.** На одном листе декомпозиции туннель не может быть одновременно произведён и потреблён. Если возникает такой случай — это нормальная внутренняя связь через X-механизм, не туннель.

### Правила именования

7. **Корень модели.** Корневая активность проекта — всегда `A0`. Любой другой id как корень — ошибка.

8. **Правильность ID.** ID активностей следуют каноничной нумерации: корень `A0`, наследники образуются суффиксами `[1-9a-z]+` (см. `01-dsl.md`). Аналогично стрелки следуют каноничной форме `[IOCMXT][1-9a-z]+`. Две градации нарушения:
   - **Structural violation** (ошибка) — id не разбирается ни в каноничную, ни в «well-formed» форму: лишний символ, цифра `0` в суффиксе, lowercase префикс, и т.п. Примеры: `A_1`, `A10`, `i1`, `O0`.
   - **Case violation** (предупреждение) — id структурно валиден, но суффикс содержит заглавные буквы вместо строчных. Примеры: `AABC` (правильно `Aabc`), `IABC` (правильно `Iabc`), `X1A` (правильно `X1a`). Не блокирует обработку, но визуально путает префикс-роль и суффикс-путь, поэтому валидатор подсвечивает.

9. **Уникальность ID.** В пределах проекта каждый ID активности встречается ровно один раз. Два файла с одним ID — ошибка коллизии.

10. **Совпадение имени файла и ID.** ID, извлечённый из имени файла, должен совпадать с ID в шапке. Несовпадение — ошибка.

### Правила структуры проекта

11. **Маркер проекта.** Корневая папка проекта определяется наличием в ней файла `A0.*.idef0`. Этот файл обязателен.

12. **Контекстный файл.** В корневой папке проекта рядом с `A0.*.idef0` обязан лежать `A-0.*.idef0`. Отсутствие — ошибка. Файл `A-0.*.idef0` в любом другом месте проекта — ошибка.

13. **Имя проекта.** Имена папок на пути от `src/idef0/` до корневой папки проекта обязаны быть валидными идентификаторами Java-пакета: первый символ — `a..z`, последующие — `a..z`, `0..9`, `_`. Точки в именах папок на этом пути запрещены. Нарушение — ошибка.

14. **Минимальная глубина.** Между `src/idef0/` и корневой папкой проекта обязан быть минимум один уровень вложенности. `src/idef0/A0.*.idef0` (A0 непосредственно в корне сканирования) — ошибка.

15. **Запрет вложенных проектов.** Внутри корневой папки проекта на любой глубине не должно быть другого `A0.*.idef0`. Вложенный проект — ошибка.

16. **Принадлежность файла проекту.** Каждый `.idef0` файл под `src/idef0/` обязан принадлежать какому-то проекту (то есть лежать внутри корневой папки какого-то проекта). Файл-сирота — ошибка.

### Расширения

17. **Количество функциональных блоков у активности.**
    - От 0 до 9 функциональных блоков — OK.
    - От 10 до 35 функциональных блоков (с использованием суффиксов `Aa..Az`) — warning «декомпозиция превышает рекомендуемый предел в 9 элементов».
    - Более 35 функциональных блоков — ошибка (за пределами разрешённого алфавита `1..9, a..z`).

18. **Порядок секций в `activity`.** Тело `activity` состоит из двух последовательных секций: сначала **boundary section** (декларации `I*`/`O*`/`C*`/`M*`/`T*`), затем **decomposition section** (декларации `A*`). См. `01-dsl.md`, раздел «Базовая структура файла».

    Граничная стрелка, появившаяся **после** первого функционального блока в том же файле, — ошибка нарушения порядка секций. Парсер всё равно её распознает и положит в `boundary[]`, но валидатор обязан репортить.

    Это правило отделяет «грамматически валидный, но семантически разбитый файл» от «грамматически невалидного». Грамматика line-based (см. `01-dsl.md`) не запрещает физический порядок — за порядок отвечает валидатор.

### Правила inherit-ID

19. **Description-identity.** Каждый литерал-описание стрелки, повторяющийся между уровнями (boundary потомка эхо-объявляет стрелку, owner которой выше), обязан **символ-в-символ** совпадать с описанием на owner-уровне. Owner определяется по таблице «Owner-таблица описаний» выше. Применяется к:
    - стрелкам в boundary активности (echo от boundary предка или от produced-list предка),
    - туннелям в boundary активности (echo от объявления в `A-0`),
    - заголовку `activity A? "name"` (echo от parent block declaration),
    - заголовку `context A-0 "name"` — это первичное объявление, owner здесь же, ничего проверять не нужно.

    Mismatch — **error**. Cascade rename (auto-fix описания при изменении на owner-уровне) — out of scope MVP. Validator только репортит mismatches, фикс ручной.

20. **Tunnel-in-boundary.** Если внутри декомпозиции активности `A?` любой её inner-block (на этом уровне или глубже) упоминает туннель `T*` — в consumed-формах `I[T*]`/`C[T*]`/`M[T*]` или в produced-форме `X..[T*]` — то `A?.boundary` обязана содержать ровно **одну плоскую запись** `T* "..."` (без префикса роли). Применяется на всех уровнях, включая `A0`. Описание этой записи проверяется по правилу 19 (owner — `A-0.context.idef0`). Отсутствие записи при упоминании туннеля внутри — error. Лишняя запись (туннель в boundary, но внутри не используется) — warning.

### Правила plug/socket

21. **Plug labels при join.** В produced list декомпозиции активности подсчитывается число плагов, подключённых к одному и тому же сокету — группировка по паре `(kind, mappedTo)`, где `kind ∈ {boundary-out, tunnel-out, parent-x-mapped}` и `mappedTo` — id сокета внутри `[...]` (`O*` / `T*` / `X*`).

    - **Multiplicity = 1** (один плаг в сокет): label плага запрещён. Если в produced написано `X11[O1] "что-то"` и этот плаг единственный, ведущий в `O1` — **error**. Описание единственного плага наследуется от сокета (для O — от плоского `O*` в boundary родителя; для T — от объявления туннеля в `A-0`; для X — от own-described `X* "..."` в produced list).
    - **Multiplicity ≥ 2** (join): каждый плаг обязан иметь label, и все label в рамках одного сокета обязаны быть уникальны (символ-в-символ). Пропущенный label у любого плага при join — error. Дубль label среди плагов в один сокет — error. Содержание label не валидируется (правило отвечает только за структуру: «есть/нет, идентичность через эхо»).

    **Источник label при join.** Label плага объявляется в produced list декомпозиции родителя — это первичное объявление. Описания в файле декомпозиции дочки (соответствующего сокета в её boundary section) обязаны эхо-совпадать с label плага по правилу 19. То есть: при join каждый ребёнок видит у себя в boundary description = label своего плага, а не общее имя сокета у бабушки.

    Применяется к **outputs**: только формы `X[O*]`, `X[T*]`, `X[X*]`. Inputs/Controls/Mechanisms в нашем DSL имеют единственный источник (фактически join у них не моделируется), поэтому правило 21 их не касается.

22. **X↔O nomenclature.** ID плага в produced list декомпозиции обязан следовать конвенции:

    `X{block_id_suffix}{output_index_char}`

    где `block_id_suffix` — суффикс id функционального блока **без ведущей `A`** (то есть весь хвост `block.id.slice(1)`); `output_index_char` — символ позиции плага в produced list этого блока: `1`..`9` для индексов 1..9, `a`..`z` для индексов 10..35. Позиция определяется в **исходном** порядке списка (1-based, без учёта сортировки форматтера).

    Примеры:
    - Родитель `A0`, блок `A1`: `block_id_suffix = "1"`. produced[0] → `X11`, produced[1] → `X12`.
    - Родитель `A0`, блок `A3`: `block_id_suffix = "3"`. produced[0] → `X31`, produced[1] → `X32`.
    - Родитель `A1`, блок `A11`: `block_id_suffix = "11"`. produced[0] → `X111`, produced[1] → `X112`.
    - Родитель `A1`, блок `A1a`: `block_id_suffix = "1a"`. produced[0] → `X1a1`.

    Несоответствие — error. Это правило закрепляет связь между X-id у родителя и O-id в boundary ребёнка: последний символ X-id = индекс O в boundary дочки. То есть для `X112` у parent `A1` (block `A11`, output_index 2) дочерний файл `A11.idef0` обязан содержать `O2` (либо `O[X*]` форму с тем же output_index — см. правило 4) в boundary.

    Поскольку `output_index` определяется позицией в исходном порядке produced list, правило не зависит от сортировки форматтера и применяется до неё.

## Виды диагностики

- **Errors** — модель не валидна, валидация не проходит.
- **Warnings** — модель валидна, но есть отклонения от рекомендаций.
- **Info** — справочные сообщения (опционально).

## Выход

Валидатор выдаёт:

- Список ошибок и предупреждений с указанием файла, строки и сути.
- Код возврата: 0 при отсутствии ошибок, ненулевой при их наличии.

## Что валидатор не делает

- Не генерирует SVG, DOT или другие визуальные форматы.
- Не интерпретирует структуру подпапок внутри проекта (это зона плагина).
- Не валидирует семантическую корректность бизнес-смысла модели (только формальные правила нотации).
- Не рассматривает файлы вне `src/idef0/`.
